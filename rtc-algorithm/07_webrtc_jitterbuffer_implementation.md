# 🔬 第七章：WebRTC 工业级防抖与同步落地

> **前情提要**：第五章和第六章，我们在纸上谈兵，用最朴素的代码建立了一个基础的“蓄水池（JitterBuffer）”和一个简单的“音视频同步器”。
> 
> **本章重点**：真正跑在二三十亿台设备上的 WebRTC，面对极其险恶的真实网络，它的做法远比我们想象的要聪明、狂野得多。本章带你看看 Google 是如何把这些理论产品化的。

## 1. 架构总览：各自为战，最后会师

在真实的 WebRTC 源码中，接收端是一个极其复杂的流水线。
音频和视频的处理完全剥离，甚至不在同一个线程里。它们各自进各自的高级蓄水池，只有在最后要送给扬声器/屏幕的那一瞬间，才会被“会师中心”统一调配。

```text
       [网络层收到 RTP 包]
              │
      分发给各自的引擎
      ↙              ↘
[NetEQ魔法引擎]   [Video JitterBuffer]
 (处理音频)         (处理视频)
      ↘              ↙
       [终极音画同步器]
        (拍板决定播不播)
              │
          [外设渲染]
```

## 2. NetEQ：让音频 JitterBuffer 变成“调音师”

我们在第五章写的音频缓冲池很笨：包如果不来，我就硬生生地等着，画面和声音直接卡顿；包如果积压太多，我就疯狂往后播，声音像快进一样。

**WebRTC 发明了 NetEQ**。它不是一个简单的缓冲池，它内置了一个“百万修音师”。

### 2.1 会变魔术的决策大管家

每次声卡来要数据时，NetEQ 的大脑（Decision Logic）会看一眼当前的蓄水池容量，然后做出如下极其聪明的决定：

1. **Normal（正常出水）**：水位正常，原汁原味地把声音送给声卡。
2. **Accelerate（无感快放）**：发现水池快满了。它不会像录音机快进那样变成“花栗鼠”的高音，而是**抽出音频里最不影响听感的片段删掉**，让声音在保持原本音调的情况下，悄无声息地加速播完，把水位降下来。
3. **Expand（无感慢放）**：发现水池快干了，下一个包还没来怎么办？硬生生把现有的 10ms 声音“拉长”到 20ms！并且保持音调不变，听起来就像人说话稍微拖了个长音，完全填补了卡顿。
4. **Merge（无缝缝合）**：那个迟到的包终于来了，但刚才我已经用魔法（Expand）拉伸出一段假的声音塞给声卡了。这时候它能做**平滑淡入淡出（Cross-fade）**，把真实的包和刚才伪造的声音完美缝合，不产生咯噔的爆音。

### 2.2 核心黑科技：WSOLA 算法

为了实现上面说到的包“无级变速”且“音调不变”，NetEQ 引入了著名的 **WSOLA (Waveform Similarity Overlap-Add) 算法**。

```cpp
// WSOLA 大白话逻辑：
// 就像把一条有弹性的丝带切成很多小段，如果你想拉长，就复制其中最相似的两段，叠在一起稍微错开缝合。
class NetEqMagician {
public:
    // 水快干了，把一段 10ms 的真音频，拉伸伪造成 20ms 的音频
    AudioFrame StretchOut(const AudioFrame& real_audio) {
        // 1. 在这段声音里找两个波形高低起伏极其相似的切点 (互相关最大)
        int best_pos = FindMostSimilarWaveformPattern(real_audio);
        
        // 2. 把这两段音频叠在一起，采用"淡入淡出"的方式拼接到一起
        // 让耳朵根本察觉不到这里被偷偷塞进去了一段复制品
        return ApplyOverlapAdd(real_audio, best_pos); 
    }
};
```
> **成就**：这套 NetEQ 系统极度强大，是 WebRTC 在各种烂网下依然保证语音通话流畅的关键护城河。

## 3. Video JitterBuffer：极其严苛的装配车间

视频 JitterBuffer 没有 NetEQ 那种“造假”的能力。视频少一块就是马赛克。
所以，视频 JitterBuffer 是一个**等级森严的装配车间**。

### 3.1 被支配的帧依赖

视频是分关键帧（I 帧，完整画面）和预测帧（P/B 帧，只记录差异）的。

```text
正常顺序： [I0帧] ←依赖 [P1帧] ←依赖 [P2帧] ←依赖 [P3帧]

如果在网络中，组装 P2帧 的某个包丢了，导致 P2 破损了。
那么就算后面的 P3、P4 完好无损地送到了，视频 JitterBuffer 也绝对不能把它们放给解码器！
因为没有 P2，即使强行解码 P3 出来的也是绿屏和扭曲的人脸。
```

### 3.2 真实代码里是怎么做的？

WebRTC 维护了一张**依赖关系图（Dependency Graph）**。每个送入 JitterBuffer 的帧，就算组装完整了，也要先经过“政审”。

```cpp
class VideoJitterBuffer {
    std::map<uint32_t, EncodedFrame> frames_; // 蓄水池
    std::set<uint32_t> decoded_;              // 记录已经成功解码的名单
    
public:
    // 只有状态变成 kDecodable（可解码的），才能放行
    void UpdateDecodability(EncodedFrame& frame) {
        if (frame.is_keyframe) {
            frame.state = kDecodable; // I帧是天选之子，自己就能过关
            return;
        }
        
        // 普通 P 帧，必须查它依赖的前辈是不是已经成功解码了？
        for (uint32_t ref : frame.依赖列表) {
            if (!decoded_.count(ref)) {
                return; // 前辈还没解码，给我老老实实在水池里等着！
            }
        }
        
        frame.state = kDecodable; // 依赖满足，放行！
    }
};
```

## 4. 救场手段：NACK 的艺术博弈

在视频车间里，如果某个关键包迟迟不到，导致一串 P 帧卡在水池里出不来，怎么办？
**呼叫发送端重传（NACK）**。

但在什么时候呼叫是有讲究的：
1. **不能立刻叫**：刚发现包没到就发 NACK，万一它只是去别的路由器绕了一下，晚半个 RTT 自己到了呢？你立刻发重传，那网络里就多了一份垃圾。
2. **不能放弃叫**：一直等，那后续所有帧都被堵死（卡顿时间太长）。

```cpp
// WebRTC NACK 调度策略：引入 RTT 感知
void OnPacketLossDetected(uint16_t lost_seq) {
    // 丢包了？先别急。
    // 计算一下这个包按理说还要多久能跑完半个网络来回 (RTT/2)
    int64_t wait_time = current_rtt_ / 2;
    
    // 定一个闹钟，如果到了 wait_time 它还没奇迹般地自己跑到手里，
    // 再果断向发送端发射 NACK 导弹。
    ScheduleNackRequest(lost_seq, Now() + wait_time);
}
```
**指数退避**：如果 NACK 发了一次，包还是没来，下一次求救的时间间隔就要翻倍（防止在拥塞时大家互相用 NACK 邮件轰炸把网络彻底挤爆）。

## 5. AV 同步终极校验：防时间漂移

第六章我们写了一个靠一次 RTCP SR 来计算音视频偏差的逻辑。
但在工业环境，**时钟是会随天气、温度和运行时间越跑越偏的（时钟漂移 Clock Drift）**。

一台在黑龙江户外跑着的手机 48kHz 声卡晶振，和一台在海南机房跑着的服务器，一天下来时间可能会劈叉几百毫秒。

**WebRTC 的真计算法：多点插值线性回归**

WebRTC 从来不只信赖“最新的一个” RTCP SR 信使。它会把过去收到的一打 RTCP SR 都收集起来。
利用算法：在二维图表上，横轴画着实际的 NTP 绝对时间，纵轴画着 RTP 相对步数。把这些点标出来，用**最小二乘法（Linear Regression）**强行拟合出一条直线。

只有这条倾斜的直线，才能过滤掉单个 RTCP 信使送货时因为网络抖动本身带来的误差，得出音视频时间映射**无穷接近真实的漂移斜率**。

然后把它交给“音为主轴的同步器”，只要发现画音偏差超过容忍阈值，通过修改视频在 JitterBuffer 里的停留时间，让画面等等声音，或加速跳过，时刻保持唇音严丝合缝。

## 6. 终局总结：一图看懂 RTC 收发全貌

走到这里，我们终于可以把整个 RTC 拥塞与媒体控制的七篇文章闭环了。

```text
┌─────────────────发送端─────────────────┐                 ┌────────────────接收端───────────────┐
│                                       │                 │                                   │
│  [视频编码] ─(发包过快)─┐               │                 │  ┌─[NetEQ音频修音池] ──(WSOLA变声抗卡顿)→ 🎤
│                      ▼               │    真实网络带    │  │                                   │
│                  [发送列车 Pacer] ─────┼─(乱序/丢包/拥堵)─┼──┤                                   │
│                      ▲               │                  │  │                                   │
│  [音频编码] ──────(让出带宽)┘           │                  │  └─[Video组装工厂] ─(查验帧连贯再放行)→ 📺
│                      ▲               │                  │       │            │              │
│                      │               │                  │       │            ▼              │
│       ┌──────────────┴──────────┐    │                  │       │     【AV 同步校验器】        │
│       │ GCC 拥塞指挥中心            │<───┼──(TWCC 延迟反馈)──┼───────┘   (算出真实NTP对齐，指令跳帧)│
│       │ (探究延迟梯度Trendline)   │<───┼──(RTCP 丢包反馈)──┼───────(呼叫 NACK / PLI 求救)     │
│       └─────────────────────────┘    │                  │                                   │
└───────────────────────────────────────┘                 └───────────────────────────────────┘
```

**我们用了一整套教程剖析了这一切：**

1. **带宽探索与自保**（一、二、三、四章）：我们在发送端，用 BBR 的建模、用 GCC 对延迟梯度的极度敏感（Trendline），死死掐住了发送列车的油门，保证不仅发出去，还不会把路由器撑死。
2. **抖动治理与装配**（五、七章）：在接收端，我们用 NetEQ 加音拉长，用视频工厂严格审查序列号，把混乱不堪的网络数据安抚成平顺的媒体内容。
3. **音画绑定**（六、七章）：用相对时钟和挂钟时间的映射，用残酷保音频弃视频的策略，保证人看直播和开会时不出戏。

这一切，全都藏在你每天随手接起的一个普通的微信视频通话、或是点开的一个普通在线会议背后，安静且强韧地运转着。
