# 📦 第五章：JitterBuffer（防抖缓冲）基础原理

> **前情提要**：前三章我们探讨的拥塞控制（BBR、GCC），都是**发送端**的独角戏——发送端绞尽脑汁地算出一个安全的码率，确保不会把网络撑爆。
>
> **本章重点**：网络再好，发出的包也不可能像钟表一样均匀到达。如何把一路颠簸、乱序到达的数据包，重新梳理成丝滑流畅的音视频画面？这就轮到**接收端**的大管家——**JitterBuffer** 出场了。

---

## 1. 为什么需要 JitterBuffer？

### 1.1 理想丰满 vs 现实骨感

在理想的网络中，发送端像节拍器一样，每隔 20 毫秒发一个音频包。接收端就在另一头，每隔 20 毫秒准时收到一个包，然后直接丢给音响播放。
这叫**恒定比特率，零抖动**。

但在真实的互联网中，数据包就像是在晚高峰的北京三环路上开车：
1. **排队不定时**：前面有没有车出车祸（路由器某个瞬间突然来了大量请求）？
2. **乱序到达**：虽然你先出发，但你选了一条红绿灯多的路（路由策略多路径），结果比你后出发的人先到了。
3. **无线网抽风**：进电梯了、遇到微波炉干扰了，Wi-Fi 突然卡了 50ms。

**结果就是：网络抖动（Jitter）。**

```text
发送端： [包1] ─20ms─ [包2] ─20ms─ [包3] ─20ms─ [包4]

接收端： [包1] ─5ms─ [包3](乱序!) ─50ms(卡顿)─ [包2](迟到) ─10ms─ [包4]
```

如果这时候接收端拿到什么就播什么，你听到的声音就是："喂…（撕裂音）…好（快放）…吗？"

### 1.2 引入"蓄水池"：JitterBuffer

为了解决这个问题，接收端在"接包"和"播放"之间，硬生生地插入了一个蓄水池，也就是 JitterBuffer。

**它的核心思想是：用延迟换平滑。**

```text
[网络侧] 杂乱无章、断断续续的包 
   │
   ▼
 ┌──────────────────────┐
 │ JitterBuffer 蓄水池   │
 │   [包2] [包3]        │  <-- 收包时，负责认领、排序、等待迟到的包
 │   [包4]      [包1]   │
 └──────────────────────┘
   │
   ▼ 取水阀门（定时器驱动，严格按照本地播放时钟匀速提货）
[播放侧] 每隔 20ms 稳稳拿走一个对应的包播掉
```

只要蓄水池里始终保留了一定量的水（缓冲池深度），不管网络上这根水管是突然断水 50ms，还是突然喷涌如注，底下的取水阀门都能按部就班地匀速出水。

---

## 2. 核心矛盾：缓冲池到底该建多大？

在 RTC（实时通信）场景里，建这个蓄水池面临着极其残酷的抉择：

**如果 Buffer 太小（比如只有 10ms）**
- **优点**：极低延迟。你说话对面瞬间听到。
- **缺点**：很容易**干涸（Underrun）**。网络稍微拉跨摇摆了 20ms，播放侧下来提货时发现水池没水了（期待的包还没来），音频直接卡顿。

**如果 Buffer 太大（比如 500ms）**
- **优点**：抗抖动能力拉满。网络就算罢工半秒，你的视频依然丝般顺滑。这也是为什么看 YouTube / 抖音 从来不卡，因为它们的播放器可能缓冲了几秒甚至几十秒的数据。
- **缺点**：延迟灾难。这是一场视频会议，别人说完话，你要等半年才听到反应。

**终极目标：自适应（Adaptive JitterBuffer）**。
我们需要一个随着网络状态动态伸缩的弹簧池。网络好时，我把水位降到最低（冲刺低延迟）；网络变差开始抖动时，我赶紧把水位拉高（保底防卡顿）。

---

## 3. 从零手写一个自适应 JitterBuffer

要在代码层面实现这个蓄水池，我们分两步走：先做容器，再做自动伸缩的策略。

### 3.1 第一步：基础排序容器

它首先得是一个能按照包序列号排列的有序字典。来得早的、来得晚的，各就各位。

```cpp
struct RtpPacket {
    uint16_t seq;           // RTP 序列号 (用于排序)
    uint32_t timestamp;     // RTP 时间戳 (用于对齐播放时间)
    int64_t  recv_time_ms;  // 我的网卡实际收到这个包的时间
    std::vector<uint8_t> payload; 
};

class BasicJitterBuffer {
    // 核心蓄水池：一个以序列号为 Key 自动排序的 Map
    std::map<uint16_t, RtpPacket> packets_;
    
    // 目标缓冲延迟（决定了我们要在水池里屯多久才放行）
    int64_t target_delay_ms_ = 50;  
    
public:
    // [网络侧] 无规律狂塞数据
    void PutPacket(RtpPacket pkt) {
        if (packets_.count(pkt.seq)) return; // 已经有了，扔掉
        packets_[pkt.seq] = std::move(pkt);  // 自动插入到正确的位置
    }

    // [播放侧] 按照时钟节奏，有规律地来要数据
    std::optional<RtpPacket> GetPacket(int64_t now_ms, uint16_t expected_seq) {
        if (packets_.empty()) return std::nullopt;

        auto it = packets_.begin();
        const RtpPacket& earliest_pkt = it->second;

        // 核心检查：这个包在我手里捂热乎了吗？
        // 捂的时间 = 当前时间 - 刚收到它的时间
        if (now_ms - earliest_pkt.recv_time_ms < target_delay_ms_) {
            return std::nullopt; // 不够时间，继续屯着，宁可晚点播（积蓄水位）
        }

        // 捂够了，交货
        RtpPacket result = std::move(it->second);
        packets_.erase(it);
        return result;
    }
};
```

### 3.2 第二步：预测网络抖动（自动伸缩）

刚刚代码里的 `target_delay_ms_` 是一个固定值 50ms。这很蠢。
真实的 JitterBuffer 需要在收到每个包时，计算网络的颠簸程度（Jitter），然后动态修改这个目标延迟。

**抖动估算原理：**
和我们在 GCC 里遇到的"延迟梯度"一模一样！
`接收方两次收包的时间差` 减去 `发送方两次发包的时间差`。
如果这个差值很大，说明网络像个过山车。我们把这些差值的绝对值平滑起来，就是当前的 Jitter 水位。

```cpp
class JitterEstimator {
    double current_jitter_ms_ = 0.0;
    
public:
    void Update(int64_t recv_time_ms, uint32_t rtp_timestamp) {
        // ... (此处省略一些差值计算代码，计算出 delay_variance)
        double delay_variance = std::abs(recv_delta - send_delta);
        
        // EWMA：用指数平滑去估算近期的平均颠簸程度，过滤毛刺
        current_jitter_ms_ = 0.9 * current_jitter_ms_ + 0.1 * delay_variance;
    }
    
    double GetJitter() const { return current_jitter_ms_; }
};

// ===================================
// 把 Estimator 接入 JitterBuffer 
// ===================================
void AdaptiveJitterBuffer::UpdateTargetDelay() {
    // 目标延迟应当是当前网络抖动的一个安全倍数（例如 2.5 倍以覆盖大部分突发）
    double raw_target = estimator_.GetJitter() * 2.5;
    
    // 关键伸缩策略：
    if (raw_target > target_delay_ms_) {
        // 网络突然变差，涨水要【快】！瞬间拉高阈值，哪怕画面停顿一下，防止后续疯狂丢帧
        target_delay_ms_ = raw_target; 
    } else {
        // 网络变好了，退水要【慢】！不可掉以轻心，慢慢把阈值往下降，追回低延迟
        target_delay_ms_ = 0.99 * target_delay_ms_ + 0.01 * raw_target;
    }
}
```

## 4. 视频领域的特殊挑战

上面讲的逻辑在处理音频时非常完美（因为音频往往一个 RTP 包就是 20ms 的声音，完全独立）。
但到了视频世界里，一切都变了。

一帧 1080P 的关键帧（I 帧）可能有几百 KB，会被切碎成几十上百个 RTP 包。
这给 JitterBuffer 带来了巨大的找茬游戏：

1. **必须组装出完整一帧才能交给解码器**。几十个 RTP 包只要丢了一个，这一整帧都是废的。如果你贸然把半成品的帧扔给解码器，画面上就会出现大块的绿屏、马赛克（花屏）。
2. **需要依赖关系检查**。如果是 P 帧（预测帧），你还得确保它的上一帧已经完整收到并且解码了，否则同样全屏马赛克。

因此，视频 JitterBuffer 往往比音频要复杂几个数量级，它维护的不是"包排序"，而是完整的**"帧排序与状态机（FrameBuffer/NaluBuilder）"**。

```
[ 收包 ]
   ↓
[ RTP Packet Buffer 环节 ]  <-- 检查包续不连续，是否带 Maker 标记
   ↓ (凑齐一具全尸)
[ Assembled Frame 环节  ]  <-- 检查这一完整帧，它依赖的上一帧在不在？
   ↓ (依赖也满足)
[ 解码/播放器提货     ]
```

## 5. 如果包就是丢了，怎么办？（补救策略）

就算蓄水池再怎么缓冲，也有等不到包彻底丢了的时候。当播放器下来提货，蓄水池只能摊手说没有。这时候怎么办？

1. **死等（Wait）**：
   画面定格，死死卡住不动，直到发端通过 NACK（重传请求）把丢失的包补过来。
   - 这就是为什么有时开会你的画面会卡住 1 秒，然后突然快放。
2. **跳过并假装没事（Error Concealment 错误隐藏）**：
   音频界常用手段。因为少听了 20ms 你根本察觉不到，播放器会自动把上一帧的声音复制拉长或者插值平滑敷衍过去。这叫 PLC（Packet Loss Concealment）。
   - 这也是为什么你打连麦语音，弱网时会觉得对方声音"电音"或者"发闷"。

---

> **本章提纲挈领**：  
> JitterBuffer 就像是快递代收点。网络好时，快递随到随取（低延迟）；网络抽风狂风暴雨时，代收点得腾出大仓库，把今天乱七八糟送来的件理顺了，等明天天气好了再有条不紊地交给解码器（高延迟抗抖动）。
> 它和拥塞控制，一收一发，共同构筑了 RTC 稳定流媒体的底座。
