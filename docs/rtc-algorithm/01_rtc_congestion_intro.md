# 📚 第一章：RTC 拥塞控制入门

> 渐进式学习路径：从"为什么需要拥塞控制"到"完整算法框架"

---

## 1. 为什么需要拥塞控制？

### 1.1 从一个真实问题说起

假设你在用视频会议，对方的网络带宽只有 **1 Mbps**，而你的摄像头正在以 **4 Mbps** 发送视频流。

这 4 Mbps 的数据包挤进只有 1 Mbps 出口的路由器，多出来的 3 Mbps 会怎样？

```
你（发送方）                路由器                   对方（接收方）
    │                        │                            │
    │── 4 Mbps 视频流 ──────▶│                            │
    │                    ┌───┴───┐                        │
    │                    │ 缓冲区│                        │
    │                    │ ████  │←── 积压，延迟飙升       │
    │                    │ ████  │                        │
    │                    │ ████  │                        │
    │                    │ 溢出！│── 丢包 ───────────────▶│
    │                    └───┬───┘                        │
    │                        │── 1 Mbps ─────────────────▶│
```

结果：**路由器缓冲区溢出 → 大量丢包 → 视频花屏卡顿**。

这就是"网络拥塞"。拥塞控制的核心职责，就是让发送速率**主动适配**网络的真实承载能力，而不是盲目地往网络里塞数据。

### 1.2 为什么 RTC（实时通信）更难处理？

传统的文件下载或网页请求用的是 **TCP**，TCP 内置了拥塞控制。但 TCP 的设计目标是"数据不丢"，遇到拥塞它会：

1. 重传丢失的包（耗费几百毫秒）
2. 大幅降低发送速率（锯齿式恢复）

这对视频通话完全不可接受。你打视频电话时，1 秒前的画面已经毫无价值，重传没有意义。

RTC 场景（WebRTC、直播推流）用的是 **UDP**，没有任何内置拥塞控制。好处是延迟低、灵活，坏处是必须**自己实现**拥塞控制。

| 维度 | TCP（文件传输） | RTC（视频通话） |
|------|----------------|----------------|
| 协议 | TCP | UDP |
| 延迟要求 | 无硬性要求 | **端到端 < 150ms** |
| 丢包处理 | 重传，等待 | 忍受丢包，不重传 |
| 码率控制 | TCP 自动调节 | **必须告知编码器调整** |
| 拥塞控制 | 内置 | **需要自己实现** |

这就是为什么 RTC 领域有专门的拥塞控制算法（GCC、BBR for RTC 等），也是本系列教程要讲的内容。

## 2. 核心概念

理解拥塞控制，需要先建立 6 个基础概念。

### 2.1 带宽（Bandwidth）

**通俗理解**：水管的粗细。带宽越大，单位时间内能通过的数据越多。

- 单位：**bps**（bits per second，比特每秒）
- 常见换算：1 Mbps = 1,000,000 bps；1 Gbps = 1,000 Mbps

**典型数值参考：**

| 场景 | 所需带宽 |
|------|---------|
| 标清视频通话（480p） | ~500 Kbps |
| 高清视频通话（720p） | ~1.5 Mbps |
| 超高清视频通话（1080p） | ~3~5 Mbps |
| 4K 直播推流 | ~15~25 Mbps |
| 家用宽带（国内平均） | ~100 Mbps |

**关键认知**：带宽是一条路径上的**上限**，不是固定值。网络繁忙时（比如周末晚高峰），可用带宽可能从 100 Mbps 骤降到 20 Mbps。拥塞控制要持续**探测**当前真实可用带宽，而不是假设它一直不变。

---

### 2.2 RTT（Round Trip Time，往返延迟）

**通俗理解**：你发出一封信，对方收到后立刻回信，你收到回信的总耗时。

```
发送方 ──── 数据包 ────▶ 接收方
发送方 ◀─── ACK ──────── 接收方
         ↑─────────────↑
              = RTT
```

**RTT 由三部分组成：**

```
RTT = 传播延迟 × 2  +  队列延迟  +  处理延迟

传播延迟：光速决定，北京→上海 ≈ 10ms 单向，固定不变
队列延迟：路由器里等候的时间，拥塞时急剧增大
处理延迟：设备处理时间，通常可忽略（< 1ms）
```

**典型数值参考：**

| 链路 | 典型 RTT |
|------|---------|
| 同城局域网 | 1~5 ms |
| 国内跨省 | 20~50 ms |
| 中国→美国 | 150~200 ms |
| 中国→欧洲 | 200~300 ms |
| 卫星链路（同步轨道） | **500~600 ms** |

**为什么 RTT 对拥塞控制至关重要？**

拥塞控制的反馈是有延迟的：你降低了码率，至少要等一个 RTT 才能看到网络的反应。RTT = 50ms 时系统可以快速响应；RTT = 300ms 时调节就很迟钝。算法必须考虑这个"响应延迟"。

---

### 2.3 丢包率（Packet Loss Rate）

**通俗理解**：寄出 100 封信，有几封丢了。

```
丢包率 = 丢失的包数 ÷ 发送的总包数 × 100%
```

**丢包的两种来源，性质完全不同：**

| 类型 | 原因 | 特征 | 对拥塞控制的意义 |
|------|------|------|----------------|
| **拥塞丢包** | 路由器缓冲区溢出 | 持续性，随负载增加 | ✅ 需要降码率 |
| **随机丢包** | 无线信号干扰、链路误码 | 偶发性，突发性 | ❌ 不该降码率 |

**视频质量与丢包率的关系：**

| 丢包率 | 视频表现 |
|-------|---------|
| 0~1% | 基本无感知 |
| 1~5% | 偶尔花屏，可接受 |
| 5~10% | 明显卡顿、马赛克 |
| > 10% | 视频基本不可用 |
| > 20% | 完全无法通话 |

**丢包驱动的控制逻辑（最朴素的拥塞控制思路）：**
- 丢包 > 10%：网络严重拥塞，**快速降码率**（乘法减）
- 丢包 2~10%：轻微拥塞，**维持码率**
- 丢包 < 2%：网络健康，**缓慢涨码率**（加法增）

> 这个"加法增、乘法减"的策略叫做 **AIMD**（Additive Increase Multiplicative Decrease），是所有拥塞控制算法的共同祖先，TCP Reno 1988 年就在用它。

---

### 2.4 BBR 霸主的基石：队列延迟（Queuing Delay）

**通俗理解**：如果我们把网络拥塞比作“外卖小哥排队等餐”，那么**队列延迟**，就是小哥到了饭店后，纯粹站在收银台前等红绿灯排队的“绝对时间”。

如果把网络交互的全部耗时看作一笔总账单（**RTT，往返耗时**），那么这笔账单由两部分组成：
> **`RTT = 纯裸跑的物理光速时间 (Propagation Delay) + 在路由器里排队等候的时间 (Queuing Delay)`**

**BBR 算法是如何测量并使用它的？**
BBR 最核心的哲学就是：**我们要坚决避免产生队列延迟，让队伍长度永远保持为 0！**
为了做到这一点，BBR 在运行过程中，会疯狂去试探并记录一个神奇的数字：**`min_RTT`（最近 10 秒内见过的极速跑满单圈的时间）**。

BBR 认为这个 `min_RTT` 就是物理极速时间。所以，**只要你现在的 RTT 大于了 `min_RTT`，多出来的这块肿胀的耗时，100% 就是队列延迟！**
此时 BBR 就会敏锐地拔刀报警：前方链路开始积压了，坚决不能再盲目加速发车了！

因此，**精确测量并清零“队列延迟”，正是新一代传输霸主 BBR 算法（TCP/QUIC）的最核心命脉。**

---

### 2.5 WebRTC 的解药：延迟梯度（Delay Gradient）

既然“队列延迟”这么好用，为什么专门做音视频会议的 WebRTC 祖师爷们（比如 GCC 算法），要在实时网络大乱射（UDP）中抛弃它，转而发明了“延迟梯度”？

**灵魂拷问：WebRTC 既然也能拿到 RTT，为什么不直接抄 BBR 的作业去算队列延迟？**
既然有了 RTCP/TWCC 反馈包，发送方完全能算出一个双程的 RTT。这时候就会产生一个极度烧脑的逻辑断层：**既然双向 RTT 容易受到单边拥塞的干扰，那为什么 BBR 算法用它就没问题，而 WebRTC 用它就是“挥刀自宫”？**

因为它们送的外卖，**性质完全不同！**

1. **TCP/BBR 送的是“大宗死物（文件下载/网页网页）”**
   它的死命令是“保证 100% 可靠送达”。由于它的发包逻辑是“一问一答（收不到 ACK 确认就不敢发新包）”，所以只要路上（无论是去程还是回程）有一丁点堵塞，它作为一条闭环的水管，整体流速当然要慢下来。大宗货物晚到 2 秒钟用户根本不在乎。因此，**BBR 可以毫无心理压力地将整个往返（RTT）视为一条不分彼此的整体管道属性。**
   
2. **WebRTC/GCC 送的是“极速保鲜活物（实时视频图传）”**
   视频体验是纯粹的**单向消耗品**！假设对方家里有人在用迅雷全满速上传文件（导致你的回程小纸条被堵死了，RTT 疯狂飙升），但你发过去的去程下行网络明明是一路绿灯的！
   如果在这种情况下，你仅仅因为双向指标 RTT 的升高，就吓得主动把发送端的高清画质切成了全屏马赛克……对方看着从高清突然变成碎片的屏幕一定会破口大骂！因为去程根本没堵！

**在实时音视频的铁律里：去程的拥塞锅去程背，回程的锅去程绝对不背！**

所以，想要精准调节去程的视频发包码率，必须彻底剥离回程干扰，只测纯净的“单向延迟（OWD）”。但悲剧的是，单向绝对时间也根本算不准，因为 NTP 时钟很难对齐（手机和电脑的时间没法分秒不差）。在这双重绝境下，聪明的 WebRTC (GCC) 做出了一个颠覆性的决定：

> **“抛弃糊涂账一般的双向 RTT！也放弃对单向绝对时长的测量！我们只算真正的去程队列‘加速度（Delta）’！” 这就是延迟梯度。**

延迟梯度的计算公式极其精准地**剥离了回程路线的任何干扰**，又完美抵消了时钟漂移的致命伤：
> **`延迟梯度 = (包2接收时间 - 包1接收时间) - (包2发送时间 - 包1发送时间)`**

如果你仔细看，**括号里的减法，永远是“用自己同一个手表的时间去互减”**，两台设备彼此的手表差了几个小时都无所谓了！

**物理学极简推演**：
1. **网络岁月静好（梯度 = 0ms）**
   你相隔 20ms 连发两包，对方也精准相隔 20ms 收到。说明路上排队队伍没变长也没变短。继续发！
2. **雷暴预警：队伍积压（梯度 = +5ms）**
   对方相隔 25ms 才收到。⚠️ **这说明在路上，包2比包1多排了 5ms 的队，拥塞正在发生！立即踩刹车减速！**
3. **拨云见日：队伍释放（梯度 = -5ms）**
   对方相隔 15ms 收到。🎉 绝地反击！说明排在前面的车开走了，即使还有人排队，情况在改善，准备试探加码率。

**这就是 GCC 算法的灵魂核心**：**BBR 追求把物理的“队列延迟”清零，而 GCC 则是把队列的“微积分变化率（延迟梯度）”用到了极致。** 这两位绝代双骄各自在 TCP 和 UDP 领域撑起了天下！

> **Bufferbloat 问题**：2011 年前后，研究人员发现路由器的缓冲区越来越大，导致拥塞时队长极高（几百毫秒），但迟迟不丢包。这个问题，彻底倒逼了业界全面转向“队列延迟（BBR）”与“延迟梯度（GCC）”这种极速的早期预警控制算法。

### 2.6 获取延迟子弹的后勤部队：RTCP vs TWCC

要想让上一节讲述的“延迟梯度”算得准，发送方就必须精确掌握**成百上千个包的精准到达时间**。这不仅催生了拥塞控制的进化，也逼迫底层的反馈机制大换血：

**第一代的坑爹古董：RTCP（1996年）**
接收方像个极度拖延的老头，每隔 1 秒才汇报一次：“过去一秒里一共丢了多少包呀，大概估算的 RTT 是多少”。
- **痛点**：粒度太太太太粗了！根本没人知道哪个包是哪一毫秒到的。靠这玩意儿根本算不了任何“逐包”的延迟梯度。

**第二代工业级基石：TWCC（2016年，Transport Wide Congestion Control）**
这玩意儿才是现代 WebRTC 跑满带宽却不死机的隐藏武器。接收方彻底变身成为了一个“极度严苛的流水线记账员”：
- 对**每一个**收到的包，都用微秒级别打上**精确的到达时间戳**。
- 每隔几十毫秒，就把这一本厚厚的、极度精细的到达时间账本，原封不动地砸回给发送方。

有了 TWCC 的密集弹药投喂，发送方的引擎才会“咔哒”一声齿轮咬合，完美算出每一刻发生微小震荡的“延迟梯度”。这也是为什么 WebRTC 能够在 0 丢包、0 花屏的情况下，依然通过调整码率让远端视频丝滑如水的全部魔法！

## 3. 拥塞控制的基本框架

### 3.1 控制环路

所有拥塞控制算法，无论多复杂，本质上都是同一个**反馈控制环路**：

```
┌─────────────────────────────────────────────────────┐
│                   拥塞控制控制环路                     │
│                                                     │
│   ┌──────────┐    ┌──────────┐    ┌─────────────┐  │
│   │ 信号采集  │───▶│ 状态估计  │───▶│  码率决策   │  │
│   └──────────┘    └──────────┘    └─────────────┘  │
│        ▲                                 │          │
│        │                                ▼          │
│   网络反馈                          编码器码率调整   │
│  (RTCP / TWCC)                                     │
└─────────────────────────────────────────────────────┘
```

- **信号采集**：从 RTCP 或 TWCC 反馈中提取丢包率、RTT、延迟梯度等信号
- **状态估计**：判断当前网络是"正常"、"轻微拥塞"还是"严重拥塞"
- **码率决策**：根据状态决定下一周期的目标码率
- **编码器调整**：把目标码率下发给编码器，编码器通过调整量化参数（QP）或帧率来适配

这个环路每隔 **100~200ms** 运行一次（具体取决于实现）。

---

### 3.2 以"丢包驱动"走一遍完整流程

> 以丢包作为拥塞信号，是最直观的实现方式。BBR 和 GCC 只是把"信号"替换成了更灵敏的测量维度，控制框架本身不变。

**场景设定：**
```
网络瓶颈带宽：1 Mbps
当前发送码率：2 Mbps（超出上限一倍）
预期结果：缓冲溢出 → 丢包 → 接收方反馈 → 发送方降码率 → 恢复正常
```

**第①步：信号采集（接收方 → 发送方）**

接收方每秒发送一次 RTCP RR 报告。路由器在 t=200ms 时缓冲区溢出，连续丢弃 5 个包（seq=11~15）。到 t=500ms 接收方发出报告：

```
RTCP RR 携带的信息：
  fraction_lost  = 64   → 换算丢包率 = 64 / 256 ≈ 25%
  cumulative_lost = 5
  last_sr + delay_since_sr → 用于计算 RTT ≈ 80ms
```

**第②步：状态估计（发送方判断）**

```
输入：loss_rate = 25% > 高丢包阈值（10%）
结论：网络严重拥塞
```

**第③步：码率决策（AIMD 乘法减）**

```
当前码率：2,000,000 bps（2 Mbps）
AIMD 公式：new_bitrate = current × (1 - 0.5 × loss_rate)
          = 2,000,000 × (1 - 0.5 × 0.25)
          = 2,000,000 × 0.875
          = 1,750,000 bps（1.75 Mbps）
```

**第④步：执行——通知编码器**

发送方将目标码率 1.75 Mbps 下发给编码器。编码器通过提高量化参数（QP 值提高 → 画质略降）或降帧率（30fps → 25fps）来减少实际发送量。

**完整时序图：**

```
发送方                     路由器（1Mbps 出口）         接收方
  │                             │                        │
  │── RTP @ 2 Mbps ────────────▶│                        │
  │                        [t=200ms]                     │
  │                        缓冲溢出，丢弃 seq=11~15       │
  │                             │── 1 Mbps ─────────────▶│
  │                             │                        │
  │◀──── RTCP RR(loss=25%, rtt=80ms) ─────────────────── │ t=500ms
  │                             │                        │
  │ 状态估计：严重拥塞            │                        │
  │ 码率决策：1,750,000 bps      │                        │
  │ 通知编码器降码率              │                        │
  │                             │                        │
  │── RTP @ 1.75 Mbps ─────────▶│                        │
  │                        [队列清空，不再丢包]            │
  │◀──── RTCP RR(loss=0%, rtt=60ms) ──────────────────── │ t=1500ms
  │                             │                        │
  │ 状态估计：正常               │                        │
  │ 码率决策：× 1.08 → 1,890,000 bps（AIMD 慢涨）        │
```

**对应的伪代码框架：**

```cpp
// ============================================================
// 基于丢包的简明拥塞控制器（教学用伪代码）
// ============================================================

class LossBasedController {
public:
    // 每次收到 RTCP 反馈时调用
    void OnRtcpFeedback(float loss_rate, int64_t rtt_ms) {
        // 步骤1：平滑处理（防止偶发抖动触发误判）
        smoothed_loss_ = 0.9 * smoothed_loss_ + 0.1 * loss_rate;

        // 步骤2：根据丢包率判断状态，决定新码率
        int64_t new_bitrate;
        if (smoothed_loss_ > kHighLossThreshold) {       // > 10%：严重拥塞
            // AIMD 乘法减：快速降速
            new_bitrate = current_bitrate_ * (1.0 - 0.5 * smoothed_loss_);

        } else if (smoothed_loss_ > kLowLossThreshold) { // 2~10%：轻微拥塞
            // 维持当前码率，等待观察
            new_bitrate = current_bitrate_;

        } else {                                          // < 2%：网络正常
            // AIMD 加法增：缓慢探测上限
            new_bitrate = current_bitrate_ * kIncreaseRate;  // × 1.08
        }

        // 步骤3：限制在合理范围内，下发给编码器
        current_bitrate_ = Clamp(new_bitrate, kMinBitrate, kMaxBitrate);
        encoder_->SetBitrate(current_bitrate_);
    }

private:
    static constexpr float   kLowLossThreshold  = 0.02f;   // 2%
    static constexpr float   kHighLossThreshold  = 0.10f;   // 10%
    static constexpr double  kIncreaseRate        = 1.08;   // 每次 +8%

    static constexpr int64_t kMinBitrate = 100'000;         // 最低 100 Kbps
    static constexpr int64_t kMaxBitrate = 10'000'000;      // 最高 10 Mbps

    int64_t current_bitrate_ = 1'000'000;  // 起始 1 Mbps
    float   smoothed_loss_   = 0.0f;
    Encoder* encoder_        = nullptr;
};
```

**三条核心规律：**

```
① 降速要快（乘法减，× 0.75~0.875）：拥塞时不能温吞吞，慢了会持续恶化
② 涨速要慢（加法增，+8%/次）：防止刚缓解又立刻触发新一轮拥塞
③ 容忍小丢包（< 2% 不响应）：网络本身存在随机底噪，不能过度敏感
```

**局限性（推动了后续算法的出现）：**

```
✗ 已经丢包才知道拥塞，响应滞后约 1 个 RTT 以上
✗ RTCP RR 每秒才汇报一次，粒度粗，无法感知短时突发
✗ 随机丢包（如 Wi-Fi 干扰）与拥塞丢包无法区分，容易误判
  ↓
  BBR：改变信号来源，直接测量带宽和 RTT，不依赖丢包
  GCC：改变反馈粒度，用 TWCC 逐包延迟梯度，在丢包前提前预警
```

## 4. 拥塞控制算法演进全景

拥塞控制从 1988 年发展至今，分为两条并行的主线：**TCP 主线**（追求高吞吐、公平性）和 **RTC/UDP 主线**（追求低延迟、实时性）。两条线问题驱动不同，但核心思路不断相互借鉴。

---

### 4.1 为什么需要拥塞控制算法？

1986 年，互联网发生了第一次**"拥塞崩溃"**事件：ARPANET 的吞吐量从 32 Kbps 骤降至 40 bps，整个网络接近瘫痪。

根因：所有主机无节制地发包，路由器丢包后主机继续重传，形成恶性循环。  
结论：必须在发送方**主动感知拥塞并降速**，而不是让路由器单方面丢弃。

1988 年，Van Jacobson 提出 AIMD（加法增加、乘法减少）和慢启动机制，奠定了此后三十年所有拥塞控制算法的理论基础。

---

### 4.2 TCP 主线：从"活下去"到"用满带宽"

#### TCP Tahoe（1988）——确立基本框架

**解决的问题**：互联网拥塞崩溃，需要最小可行的稳定方案。

**核心创新：**
- **慢启动（Slow Start）**：不是"慢"，而是"指数增长探测带宽"。初始发 1 个包，确认后发 2 个，再确认发 4 个……直到感到拥塞为止。
- **AIMD**：正常时线性增加窗口（+1/RTT），丢包时窗口直接归 1 重新慢启动。
- **拥塞窗口（cwnd）**：用"水桶"限制在途包数，水桶大小 = cwnd × MSS。

**局限**：丢包后窗口归 1，恢复极慢；把所有丢包都当拥塞（随机丢包会误判）。

---

#### TCP Reno（1990）——加速丢包恢复

**解决的问题**：Tahoe 丢包后恢复太慢，带宽利用率低。

**核心创新：**
- **快速重传（Fast Retransmit）**：收到 3 个重复 ACK 立即重传，不等超时。
- **快速恢复（Fast Recovery）**：丢包后窗口减半（cwnd / 2），进入线性增长，不再重新慢启动。

**效果**：丢包恢复从"归零重来"变成"打折继续跑"，吞吐量明显提升。  
**局限**：窗口减半是固定比例，对高带宽延迟积（BDP 大的长胖网络）不友好。

---

#### BIC TCP（2004）——解决高带宽场景

**解决的问题**：千兆网络普及后，TCP Reno 的线性增长根本填不满高 BDP 链路的带宽。

**核心创新：二分搜索找带宽上限**
- 记录上次丢包时的窗口 `W_max` 作为上限参考
- 用二分搜索在 [当前窗口, W_max] 之间快速逼近上限
- 接近上限时放慢增速，远离上限时加快增速

**效果**：在高 BDP 场景（如 10 Gbps × 100ms RTT）吞吐量比 Reno 提升 10 倍以上。  
**局限**：二分搜索的增长曲线对短 RTT 链路过于激进，公平性较差。

---

#### CUBIC TCP（2008，Linux 默认至今）——三次曲线平滑增长

**解决的问题**：BIC 增长曲线太激进，在低 BDP 场景不友好。

**核心创新**：用三次函数替代 BIC 的二分步骤：

```
W(t) = C × (t - K)³ + W_max
K = ∛(W_max × β / C)     （β ≈ 0.7，丢包后的窗口缩减比例）
```

- 刚丢包后恢复快（距 W_max 远，曲线陡）
- 接近 W_max 时增速缓（避免立刻再触发拥塞）
- 超过 W_max 后继续缓慢探索新上限

**效果**：高低 BDP 场景均表现稳定，成为 Linux 内核默认实现。  
**局限**：本质上仍"等到丢包再反应"，Bufferbloat 场景下 RTT 会膨胀到几百毫秒。

---

#### BBR v1（2016，Google 提出）——彻底换思路

**解决的问题**：
1. Bufferbloat 让路由器缓冲区越来越大，CUBIC 填满缓冲才丢包，RTT 膨胀数百毫秒。
2. 无线随机丢包让基于丢包的算法频繁误降速。

**核心创新：不再等丢包，直接测量两个物理量**
- **BtlBw**（瓶颈带宽）：最近一段时间测得的最大吞吐量。
- **RTprop**（最小 RTT）：最近 10 秒内的最小 RTT（代表空队列时的传播延迟）。

目标码率 = BtlBw，让在途数据量 ≈ BtlBw × RTprop（不把多余数据填入缓冲区）。

**效果**：
- 高丢包场景（10% 随机丢包）吞吐量比 CUBIC 高 **100 倍以上**（YouTube 实测数据）。
- RTT 从数百毫秒降低到接近真实传播延迟。

**局限**：多 BBR 流竞争时公平性差；ProbeBW 阶段会造成瞬时队列积累。

---

#### BBR v2（2019）& BBR v3（2023）——修复公平性

**解决的问题**：BBR v1 与 CUBIC 共存时会挤占 CUBIC 的带宽，多 BBR 流之间也不公平。

- **BBR v2**：引入丢包信号辅助，检测到丢包时降低 inflight 上限，与 CUBIC 竞争时更友好。
- **BBR v3**：进一步精细化 ProbeBW 探测策略，改善突发行为，是目前最新的 Linux 实现。

---

### 4.3 RTC/UDP 主线：从"能用"到"体验好"

RTC 场景（视频通话、直播）使用 UDP，不能依赖 TCP 内置的拥塞控制，必须自己实现。

```
1996          2010          2012          2016              2021
 │             │             │             │                 │
RTCP RR      GCC v1        NADA         GCC v2 / GOOG-CC   持续演进
(RFC 3550)   延迟梯度雏形   IETF RMCAT   WebRTC 标准默认
                           SCReAM（爱立信）
```

---

#### GCC v1（~2010，Google 内部）——第一个实用 RTC 拥塞控制

**解决的问题**：视频通话（Hangouts）在 UDP 上没有任何拥塞控制，体验极差。

**核心设计：双通道信号**
- **丢包通道（RTCP RR）**：检测严重拥塞，触发快速降码率。
- **延迟通道（单向延迟梯度）**：在丢包发生前就感知拥塞趋势，平滑调整。

延迟梯度用卡尔曼滤波平滑噪声，再用阈值状态机（Overuse / Normal / Underuse）决定升降速。

**意义**：第一次在 RTC 场景中用"延迟梯度"作为主要拥塞信号，比丢包提前 1~几个 RTT 感知拥塞。

---

#### NADA（2012，Cisco，IETF RMCAT 提案）——学术严格化

**NADA = Network-Assisted Dynamic Adaptation**

**核心创新**：把丢包、延迟、ECN（路由器显式拥塞通知）三种信号统一转化为一个"虚拟队列延迟"值：

```
x_n（虚拟延迟）= 实测延迟梯度 + 丢包惩罚项 + ECN 惩罚项
```

用统一的 x_n 驱动线性控制律（PID 风格）调整码率，而不是离散的状态机。

**意义**：理论上更优雅，对 ECN 网络支持更好；但现实中 ECN 普及率低，影响力有限。

---

#### SCReAM（2016，爱立信）——专为超低延迟设计

**SCReAM = Self-Clocked Rate Adaptation for Multimedia**

**核心创新：自时钟（ACK Clocking）发送**
- 用 ACK 到达节奏来决定下一个包的发送时机，天然适配网络节奏，不依赖外部定时器。
- 在应用层维护"虚拟队列"，通过控制队列大小来约束延迟。

**适用场景**：RTT < 20ms 的超低延迟场景（游戏语音、远程控制），在此场景下优于 NADA 和 GCC。

---

#### GCC v2 / GOOG-CC（2016，Google，WebRTC 标准默认）——工业级成熟

**解决的问题**：GCC v1 反馈粒度粗（RTCP 每秒 1 次），延迟梯度计算不精确。

**关键升级：**

1. **引入 TWCC**：反馈从 RTCP RR（1秒1次，统计均值）升级为 TWCC（每 50~100ms 一次，逐包精确时间戳）。

2. **带宽估计分两层**：
   - **延迟估计器（Trendline Filter）**：用线性回归替代卡尔曼滤波，判断延迟趋势，是主通道。
   - **丢包估计器**：丢包 > 10% 时介入，做安全兜底。
   - 最终码率 = min（延迟估计器结果, 丢包估计器结果）。

3. **ALR（应用层码率限制）**：编码器产生数据不足时，避免过度探测带宽（探测会增加延迟）。

**现状**：Chrome、Edge、Firefox 的 WebRTC 均默认使用 GOOG-CC，是目前全球使用最广泛的 RTC 拥塞控制实现。

---

#### BBR on QUIC / WebTransport / RTP over QUIC（2019 至今）——BBR 进入 RTC 场景

你可能会问：BBR 不是 TCP 的吗？RTC 的 UDP 能用 BBR 吗？

答案是可以，路径有三条，成熟度各不同：

**① QUIC + BBR（已大规模部署）**

QUIC 是 Google 设计的传输层协议，运行在 **UDP** 之上，是 HTTP/3 的底层。QUIC 的拥塞控制是**可插拔的**，Google 默认使用 **BBR**。

影响范围：
- 你访问 Google、YouTube 时，大概率已经在用 QUIC + BBR
- 全球 HTTP/3 流量占比已超过 25%（2024 年数据）
- 但这是纯数据传输（文件、网页），不是实时音视频

**② WebTransport over QUIC（2023 年标准化）**

WebTransport 是 W3C/IETF 联合推进的新传输 API，底层是 QUIC，因此天然继承 BBR。它支持：
- 双向流（Streams）：类似 WebSocket，但基于 QUIC
- 数据报（Datagrams）：类似 UDP，但有拥塞控制（BBR）

应用方向：WebTransport Datagrams 可以承载实时音视频，作为 WebRTC DataChannel 的下一代替代。目前 Chrome 已支持，处于早期采用阶段。

**③ RTP over QUIC（IETF 草案，尚未标准化）**

IETF AVTCORE 工作组正在推进 [draft-ietf-avtcore-rtp-over-quic](https://datatracker.ietf.org/doc/draft-ietf-avtcore-rtp-over-quic/)，目标是把 RTP 音视频媒体流直接跑在 QUIC 上。这样 WebRTC 的音视频就能用 QUIC 内置的 BBR 作为拥塞控制，彻底替换 GOOG-CC。

**现状**：草案仍在讨论中，尚无主流实现。

**三条路径对比：**

| 路径 | 传输层 | 拥塞控制 | 成熟度 | 适用于音视频 |
|------|--------|---------|--------|------------|
| QUIC + BBR | UDP（QUIC） | BBR | ✅ 大规模部署 | ❌ 主要用于数据 |
| WebTransport Datagrams | UDP（QUIC） | BBR | 🔶 早期采用 | ✅ 可承载媒体 |
| RTP over QUIC | UDP（QUIC） | BBR | 🔴 草案阶段 | ✅ 专为媒体设计 |
| 现有 WebRTC | UDP（裸 RTP） | GOOG-CC | ✅ 主流标准 | ✅ 当前最成熟 |

> 长期来看，RTP over QUIC 标准化后，BBR 很可能成为 RTC 音视频的主流拥塞控制，GOOG-CC 的延迟梯度思路会以某种形式被融合进来。

### 4.5 核心驱动力总结

| 驱动问题 | 产生的算法 |
|---------|-----------|
| 拥塞崩溃，网络无序 | TCP Tahoe（AIMD + 慢启动） |
| 丢包后恢复太慢 | TCP Reno（快速重传 + 快速恢复） |
| 高带宽场景填不满 | BIC → CUBIC（二分 / 三次曲线） |
| UDP 上没有拥塞控制 | GCC v1（延迟梯度 + 双通道） |
| Bufferbloat / 随机丢包误判 | BBR v1（直接测带宽 × RTT） |
| RTC 需要逐包精确反馈 | TWCC → GCC v2 / GOOG-CC |
| BBR 多流不公平 | BBR v2 / BBR v3 |
| 超低延迟场景（RTT < 20ms） | SCReAM（自时钟） |
| 数据传输 / 媒体想用 BBR on UDP | QUIC+BBR → WebTransport → RTP over QUIC |

> 本系列后续章节将深入讲解 **BBR**（第2章）和 **GCC/GOOG-CC**（第3章），这两个算法代表了 TCP 和 RTC 各自主线上当前最成熟的工业实现。
