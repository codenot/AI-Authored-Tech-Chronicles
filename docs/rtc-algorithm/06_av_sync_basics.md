# 🎯 第六章：音视频同步基础（AV Sync）

> **前情提要**：第五章里我们为音频和视频各自建立了一个 JitterBuffer 蓄水池，把颠簸的包梳理平整了。
> 
> **本章痛点**：音频池子和视频池子是各管各的！如果音频阀门放水快了，视频阀门放水慢了，或者网络导致视频晚到了半秒被拦在了蓄水池外——结果就是灾难性的**“画音不同步”**：你看到别人嘴唇动完了，过了一秒才听到声音。
>
> **本章目标**：如何把这两股本来毫不相干的数据流，牢牢绑定在同一条时间线上？

---

## 1. 为什么“口型”总会对不上“声音”？

在 RTC 传输的全链路上，音频和视频简直就像是在两条截然不同的赛道上蒙眼狂奔，它们失步的原因太多了：

1. **采集端的时钟不同**：
   - 麦克风按 48000Hz 采音，摄像头按 30FPS 采图。这两个硬件是由主板上不同的晶振驱动的，天然就有微小的速度差（时钟漂移）。
2. **编码耗时天差地别**：
   - 压一帧音频几乎瞬间完成（< 1ms）。
   - 压一帧 4K 视频可能要耗费 10~20ms。视频天生“起步”就比音频慢。
3. **网络路由策略不同**：
   - 音频包小且重要，可能会被路由器优先转发（QoS 较高）。
   - 视频包巨大，塞在队列里慢慢挤。
4. **接收端解码与缓冲的差异**：
   - 音频拿出来就能播。视频还得等完整的一帧拼好、并且参考帧也到了才能解，视频 JitterBuffer 的“出水”往往比音频被动得多。

---

## 2. 人类对“声画错位”的容忍度

如果我们做不到绝对的 0 毫秒同步误差，那么允许偏差多少？
一个极其重要的生理常识是：**人类大脑对“声音超前”和“声音滞后”的容忍度是不对称的！**

- **声音比画面先出来（音频超前）**：
  - > 45ms，人就会极其难受！因为在现实物理世界中，光速远大于声速，**绝不可能出现“先听到声音，后看到嘴动”的现象**。一旦出现，大脑的潜意识就会疯狂报警，觉得极其违和。
- **声音比画面晚出来（音频滞后 / 视频超前）**：
  - < 125ms，人都可以勉强接受。因为现实中我们看远处的闪电、看远处的工地打桩，都是先看到动作，后听到声音的。大脑习惯了声音稍微晚一点到。

> **策略心法**：宁可让视频画面等声音（视频卡一下），也**绝对不能**让声音跑到了画面前面！

---

## 3. 同步的基石：三把尺子

既然音视频是两条独立的流，要怎么把它们对齐？我们需要三把关键的尺子：

### 第一把尺：RTP 时间戳（内部节奏尺）

音频和视频各自的 RTP 包头里都有一个 timestamp。
但**这把尺子是相对的，且单位不一样**：
- 音频的单位是采样数（比如 48kHz 下，1秒走 48000 步）。
- 视频的单位是 90kHz 时钟（1秒走 90000 步）。

如果音频 RTP 是 480、视频 RTP 是 900，你根本算不出它们是不是同一时刻发生的事，因为它们的“起点（0刻度）”都是发车时随机生成的。

### 第二把尺：NTP 时间（现实世界绝对时间的挂钟）

我们需要一把现实世界的绝对时间尺（比如东八区 19:00:00.000）。NTP 就是这个绝对时间的表示格式（64位整数）。

### 第三把尺：RTCP SR（红娘/桥梁）

怎么把 RTP 相对时间，翻译成现实绝对时间？
发送端会定期（一般几秒钟一次）向接收端发送一种特殊报文——**RTCP SR（Sender Report，发送者报告）**。
它就像一个“快照”，里面写着：
> “喂！刚才发过去的按 **音频 RTP=48000** 那个包裹，它对应的现实时间是 **19:00:01.000**！”
> “喂！刚才发过去的按 **视频 RTP=90000** 那个包裹，它对应的现实时间是 **19:00:01.000**！”

有了这两个 RTCP SR 信使，接收端就恍然大悟：哦！原来音频的 48000 和视频的 90000 是同时被录下来的！它们在绝对时间轴上终于被对齐了。

## 4. 极致简化的 AV Sync 对齐推演

假设现在接收端准备往屏幕上刷一帧视频了，它该怎么判断是不是同步的？

```cpp
// 1. 我们收到了最新的音、视频 RTCP SR，建立了映射表
RTCPSenderReport audio_sr; // {"RTP_A: 48000" == "NTP: 19:00:01"}
RTCPSenderReport video_sr; // {"RTP_V: 90000" == "NTP: 19:00:01"}

// ... 播放器正在运行中 ...

// 2. 拿到当前准备播放的包的 RTP 时间戳
uint32_t current_audio_rtp = 取出喇叭正在播的音频包_RTP(); // 假如是 48480 (过了10ms)
uint32_t current_video_rtp = 取出准备刷到屏幕的视频帧_RTP(); // 假如是 92700 (过了30ms)

// 3. 把它们翻译回发送端的绝对挂钟时间
int64_t audio_wall_ms = RtpToNtpTime(current_audio_rtp, audio_sr); // 算出来是 19:00:01.010
int64_t video_wall_ms = RtpToNtpTime(current_video_rtp, video_sr); // 算出来是 19:00:01.030

// 4. 计算偏差！
// 音频时间 - 视频时间 = 10ms - 30ms = -20ms
int64_t offset_ms = audio_wall_ms - video_wall_ms; 
```

`offset_ms = -20ms` 意味着什么？
意味着：目前音响里播出来的声音（早前 10ms 录的），比屏幕上正要显示的画面（早前 30ms 录的）**旧了 20ms**。
或者说：**视频画面跑到了声音的前头（超前 20ms）！**

---

## 5. “以谁为主”的博弈树

算出了偏差 `offset_ms`，接下来该怎么纠偏？有三种选择：

### 5.1 方案 A：让音频去迎合视频（极其糟糕）
"声音变慢一点，或者跳过一段声音，等画面追上来"。
**❌ 致命缺陷**：人耳对声音的连续性极其敏感。你就算少传一张图，我也能猜出动作；但你敢把一首正在放的歌加速 1.2 倍、或者中间消音 20 毫秒，那刺耳的撕裂感会直接让人抓狂。

### 5.2 方案 B：各凭本事，都认绝对时钟（勉强能用）
分别算好这帧音频、这帧视频对应的精确本地时间，只要这个时间一到，就往外丢。
**❌ 缺陷**：受本地系统时钟精度的干扰，且没有解决两端时钟由于晶振误差带来的长线“漂移”问题（打个半小时的电话，时间轴直接劈叉）。

### 5.3 方案 C：以音频为主，生杀大权交给视频（WebRTC 的终极答案）

**Audio Master 同步策略的核心思想**：
音频 JitterBuffer 和声卡就是大爷。它们以恒稳的步伐往解码器里送音频，不受任何其他流干扰。
所有的妥协，**全部由视频来承担**！

```cpp
class AudioMasterSyncStrategy {
public:
    // 每当视频准备渲染下一帧时，来问问这个管家
    Action CheckVideoFrame(int64_t audio_wall_time, int64_t video_wall_time) {
        int64_t offset = audio_wall_time - video_wall_time;
        
        if (offset > 40) { 
            // 声音跑在了画面前面太多 (>40ms)！
            // 决不允许让声音等画面，必须让画面加速跑起来追上！
            // 动作：跳过眼下这帧视频（Drop），赶紧把下一帧刷出来
            return kDrop; 
        } 
        else if (offset < -60) {
            // 画面跑在了声音前面太多 (>60ms)！
            // 动作：按住画面别动（Hold），等声音追上来再放行
            return kHold;
        } 
        else {
            // 在容忍区内 (-60ms ~ 40ms)，安全！
            return kRender;
        }
    }
};
```

这就叫**“弃卒保车”**。在极端恶劣的网络下，WebRTC 的策略永远是：哪怕视频卡成 PPT、甚至视频干脆几秒不刷新，也必须确保你的音频尽可能连贯和实时。

> 这种以“听感”为第一顺位的决策，是做实时声网技术的金科玉律。
