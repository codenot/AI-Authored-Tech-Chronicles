# OpenSpec 框架使用规范与最佳实践

OpenSpec 是一种轻量级、跨模型的 AI 协同工作流协议，旨在通过文档化的方式固化架构决策，避免 AI 编程中的上下文丢失与规范漂移。本文档整理自官方推荐原则，提供标准操作步骤与最佳实践。

## 一、 核心工作流规范

OpenSpec 强制执行“提案-执行-归档”的三步工作流。所有涉及架构或核心逻辑修改的动作，都必须遵循此流程。

### 1. 提案生成 (Propose)
在编写或修改任何代码之前，首先要求 AI 在项目根目录生成 `proposal.md` 文件。
- **目标**：明确修改动机与架构边界。
- **最佳实践**：
  - 必须包含“Why（修改动机）”以及“Constraints（绝对不能违背的红线）”。
  - 约束条件需具体量化（例如：延迟限制 `< 5ms`，内存占用 `< 50MB`，禁止特定的 I/O 操作）。

### 2. 规范执行 (Apply)
AI 必须主动读取并严格遵循 `proposal.md` 中的约束条件进行工程实施。
- **目标**：确保代码实现与架构提案保持绝对一致。
- **最佳实践**：
  - 每次执行模块编写时，通过指令显式要求 AI 挂载并参考该提案。
  - 如代码生成结果偏离规范（如引入未经批准的外部依赖），应立即依据提案进行纠偏。

### 3. 持久化归档 (Archive)
代码验证通过并合入主干后，对根目录的 `proposal.md` 进行归档处理。
- **目标**：形成项目的架构演进日志，供未来开发或新 AI 会话读取。
- **最佳实践**：
  - 将 `proposal.md` 重命名为带有时间戳和功能描述的文件（如：`20260327_auth_plugin_refactor.md`）。
  - 统一移动并妥善存放在项目根目录下的 `delta-specs/` 文件夹中。

## 二、 提案文档模板参考

一份标准的 `proposal.md` 应保持专业紧凑，推荐采用以下结构：

```markdown
# [功能或模块名称] 提案

## 修改动机 (Why)
- [列出业务背景、痛点及预期要达成的目标]

## 架构约束 (Constraints)
1. **[性能限制]**：[例如：执行路径总延迟必须 < 5ms]
2. **[技术红线]**：[例如：禁止在该路径中建立新的数据库连接，仅允许读取本地内存缓存]
3. **[安全与内存规范]**：[例如：必须使用零拷贝 API 进行数据抽离]

## 实施范围 (Scope)
- [涉及需要修改的目标文件或模块列表]
```

## 三、 规范漂移 (Spec Drift) 防范

在团队协作或手动介入的场景下，往往容易导致文档与实际代码脱节，产生“规范漂移”。为避免过期的规范误导 AI，应遵守以下准则：

1. **强制反向同步**：如果开发者脱离 AI 辅助，手动对核心逻辑进行了调整，必须第一时间要求 AI 进行反向同步：
   > *“基于当前的实际代码改动，同步更新并归档最新的 `proposal.md`。”*
2. **定期审查**：对于迭代周期较长的项目，定期（如按月或随版本发布）审核 `delta-specs/` 目录，清理或合并已经失效的历史规范，确保保留有效的“架构记忆”。
