# 🚀 抛弃 `develop` 的断舍离：现代 Git 工作流的演进与权衡

> **题记**
> 是否需要引入 `develop` 分支，本质上是在权衡 **开发隔离性** 与 **交付效率**。而现代工程实践给出的答案是：引入 `develop` 往往只是“掩盖问题”，而非真正的“解决问题”。 

你可能第一天入职，老一代架构师语重心长地对你说：“我们严格按照 Git Flow 来，所有人的功能先合进 `develop` 分支，大家在这边联调。稳定了，再合进 `main` 去发版。”听起来是不是很有安全感？

但现实很快会教做人：`develop` 分支并没有变成一个整洁的过渡“缓冲区”，反而往往成了一个**永远达不到发版标准的大型垃圾填埋场**。A童鞋的代码合进去把B童鞋的改崩了；C童鞋的功能没测完，导致所有在 `develop` 里的功能都被迫延期，最后谁都不敢往 `main` 里合。

为什么会这样？这就引出了一个致命问题：在现代高频交付的节奏下，繁重的长期分支模型已经成为了最大的绊脚石。

---

## 一、 为什么我们要革 `develop` 分支的命？

让我们先为分支建立一个直观的**物理学原理映射**：
把代码合并想象成**公路上的车流**，`main` 就是双向八车道的**高速主干道**。

- **经典 Git Flow 的困局**：它的架构思想是在上高速之前，建一个巨大的**集散收费站**（`develop`）。所有的车辆（特性分支）必须先在这个收费站里挤作一团，等所有人排好队、安检完，再统一抬杆上高速。这就必然导致几十台车互相连环撞车，车流发生严重积压。
- **现代主干开发 (Trunk-Based Development)**：直接砸毁集散收费站，让你在辅路起步（`feature` 分支）的时间强制 `< 3 天`。一旦你车造好，通过自动化安检后，立刻用高架桥并入 `main` 这条高速，随时出发。

拿掉了收费站，安全感没了怎么办？这就需要引入更聪明的“交规”。

### 那些直击灵魂的恐惧

当你强势建议砍掉 `develop` 分支时，团队里的保守派一定会跳出来，这也是我们必须直面的痛点：

**“如果必须合并到 `main` 才能部署测试，合并进去后发现是个致命 Bug，导致 `main` 被污染、整个团队被卡死无法上线，这锅谁背？！”**

为了应对缺少 `develop` 带来的恐惧，现代开发为你准备了以下几个维度的“兵器库”来护航：

| 痛点恐惧 | 现代应对方案（治本策略） |
| :--- | :--- |
| **怕半成品污染主干** | 👉 放弃长期隔离，使用 **Feature Flag（特性开关）**，像魔法“隐形披风”一样把半成品藏起来。 |
| **如何维持集成测试环境？** | 👉 从 `main` 自动部署到 STAGING 环境，基于打出的临时 Tag 去测，而非依赖 `develop` 这滩浑水。 |
| **多人协作经常爆发冲突** | 👉 强逼**短生命周期分支**（存活时长绝对 `< 3天`） + **频繁 rebase/merge main**，让冲突化解在微处。 |
| **发布前需冻结缓冲** | 👉 通过打 **Release Candidate (RC) 版本标** 或 建立**极短期的 Release 分支**（几天内死掉），而非维护幽灵分支。 |

## ❓ 灵魂拷问：`develop` 和其他流派的 `main` 到底有啥区别？

> **“在物理结构上，Git Flow 的 `develop` 不就是别人家的 `main` 吗？大家都是代码的初始大本营啊！”**

这是一个极其敏锐的观察！没错，从代码结构上看，它们都是开发者合入代码的“池子”。但为何我们要将其**物理消灭**？因为这背后隐藏着截然不同的**“发布哲学”与“心理暗示”**。

让我们用**“餐厅出餐”**的直觉来解构这个差异：

- 🍲 **Git Flow（有 develop + main）：传统的封闭后厨与前厅**
  - **`develop` = 封闭的后厨（施工现场）**，**`main` = 餐厅的展示台（面向大众的现网）**。
  - 几百个厨师在 `develop` 这个后厨里疯狂做大锅饭。大家觉得“反正这是后厨，乱一点没关系，反正离端给客人还早”。菜都在后厨堆积，直到规定的“发版日（Release）”，大堂经理才把这几十道大混菜一股脑儿端去 `main` 这个展示台。
  - **致命缺点**：如果上菜前发现托盘里有一道菜是馊的（一个 Bug），那整盘菜（整个 Release）都被迫扣住，大家只能在厨房里“修仙救火”。这就是**集成地狱**。

- 🍣 **主干开发（只有 main）：开放式厨房 / Omakase (板前料理)**
  - **`main` 既是厨房的砧板，又是客人的餐桌**。
  - 没有隐藏的后厨。厨师只要切好了一块精致的生鱼片（一个短生命周期的 Feature），经过严格的试毒（CI/CD 测试），就**直接端给客人**（合入 `main` 即等于随时准备发版）。
  - **核心哲学是消除“虚假的安全感”**：如果保留了 `develop`，程序员总会觉得“没事，反正还没合主干，可以慢慢修”。而主干开发要的就是那份冷酷的压力——**“你现在合向 `main` 的代码，下一秒就可能在百万用户的手机里爆炸。所以，你最好把 Feature Flag 关好，把自动化测试拉满！”**

## 二、 后卫防线的“双煞”：Feature Flag 与 RC

把 `main` 当作唯一的真理之源，我们就必须要掌握这两种实战战术：

### 1. Feature Flag (特性开关)：“隐形披风”战术
如果一个大功能需要开发大半个月，没法在 `< 3 天` 内完工，难道不准合代码了？
只要给新加的代码套个**“物理控制阀”**。只要阀门不开，无论是用户还是测试环境都不会执行到这部分。即使代码里面藏着毁灭世界的 Bug，它也是处于静默冬眠状态的。

```cpp
// 伪代码演示：用 Feature Flag 绝对隔离危险的半成品代码
// 这个开关在远端配置中心，或者编译时由环境变量传入
bool is_new_payment_active = GetFeatureFlag("NEW_PAYMENT_V2");

if (is_new_payment_active) {
    // 🚧 魔法隐身区！
    // 哪怕 RouteToNewPaymentGateway() 把内存耗尽、不断产生死锁
    // 只要开关关着，主干 main 对所有的普通用户就是绝对安全的！
    RouteToNewPaymentGateway(); 
} else {
    // 稳定可靠的、正在经受线上考验的旧版逻辑
    RouteToLegacyPayment(); 
}
```

### 2. 构建候选版本 (Release Candidate)：告别“实时跟车”
不是每一次往 `main` 合并代码，测试同学都要吭哧吭哧跟进测试。团队必须打断这种“保姆式跟车”：
**要在每天或每轮开发结束点，从 `main` 剪切一个“代码快照”并打上临时 Tag，这就是 RC 版本 (Release Candidate)。**
发布和测试的锚点必须建立在这个特定切片上，只有这样，开发者才能继续无所顾忌地往 `main` 合并。

## 三、 GitHub Flow vs GitLab Flow vs OneFlow 的华山论剑

当我们把 `develop` 扫入历史的故纸堆后，由于大家的“开发动作”高度趋同，业界逐渐演化出了三种主流流派。
它们在**开发体验**上几乎不分伯仲：
👉 **拉取短命 `feature` 分支 ➝ 提 PR ➝ 合入 `main`。**

真正的分水岭在于：**到底该通过什么动作，让代码真正跑向用户？**

```text
+-----------------------+
|  💻 开发体验 (三者高度共识) |   无 develop，仅留 main
+-----------+-----------+
            | 
  [开辟工位] git checkout -b feature/xyz
            |
  [合并交付] 合并回 main 高速公路
            |
+-----------v-----------+
|  🚀 流量引入与发布策略  |   <-- 真正的抉择在这里！三岔路口
+-----------------------+
```

### ✅ 核心机制字典对决

| 工作流 | 核心发版思想 | 物理学类比 | 最适配的场景与权衡 |
| :--- | :--- | :--- | :--- |
| **GitHub Flow** | **持续部署：合并即上线**<br/>`main` 始终精确等于生产环境。 | **活字印刷机**：刻好一个字块，刚拼进模板，立刻就压在纸上印发。 | **SaaS、Web 应用**。<br/>一旦出事能瞬间 Rollback 回滚；用户无感知更新；不再维护旧版本包袱。 |
| **GitLab Flow** | **利用分支映射物理环境**<br/>以分支之名，行控制之实（`staging` ➝ `production`）。 | **多级分流净水器**：所有水流从 `main` 而来，经过层层滤网（分支），最终抵达龙头。 | **重度 2B 企业项目**。<br/>受制于合规审查、灰度测试，或者存在不能完全自动化的测试环节。 |
| **OneFlow** | **用 Tag 作为发布驱动力**<br/>发版只需从 `main` 狠狠打个 Tag；修 Bug 时回到 Tag 旁拉分支。 | **为疾驰的列车直接喷漆**：不需要额外铺设轨道，这趟车拉够了货，直接喷个 `V1.2` 贴花发走。 | **移动端应用 (APP)、开源库**。<br/>历史线无比干净，兼顾热修复，彻底斩断对 `release/develop` 的依赖。 |

> [!WARNING]
> GitLab Flow 和 OneFlow 中也会容忍“Release”分支的存在。但请注意：这些发布分支是极度临时的！它们的预期存活时间只有**发布窗口期的短短几天**。千万不要把它搞成了 Git Flow 里那个万年不死的“僵尸分支”。

## 四、 现实的妥协与代价 (Tradeoff)

去掉 `develop` ，转向主干开发，绝对不是改个名字这么简单。它的背面隐藏着巨大的代价与妥协：

1. **强行倒逼极致的质量门禁**
如果没有严丝合缝的验证系统，没有由无数个 Unit Case 严防死守的 PR（Pull Request），那么撤去 `develop` 这道防线后，低质量的“丧尸代码”会长驱直入、直捣 `main` 的心脏。所以，**主干开发的底气，全拜强大的 CI（持续集成）所赐**。
   
2. **对工程师开发习惯的生猛改造**
程序员再也无法理直气壮地说出：“这个模块太大了，我得等半个月全写完再提交。”代码合并频次必须从 `1次/周` 被生生掰碎成 `1次/天` 甚至是 `3次/天`。**小步快跑，高频微量**，不仅是为了减少冲突，更是为了将 Bug 的爆炸当量从“核弹级”降格到“擦炮级”。

正如我们在开头所讲，过去我们太依赖 `develop` 这种“测试温室”来掩盖我们不敢面对频繁交付的恐慌。只有主动打破这层玻璃，强迫整个团队去面对那条时时刻刻可能破碎的主干，才能逼着大家修筑出真正的钢铁防线。

## 五、 写在最后与下章伏笔

技术的发展就是一条不断做减法的极简之路。
**在现代 Git 工程哲学中，“分支”（Branch）往往代表的应该是一种**短期动作**（修复一个Bug、做一个Feature、维护一个临时发版），而不是一种物理常驻状态。** 把环境与状态的承载留给 Tag 和 CI 吧，让 `main` 成为那个唯一向前的“时间之矢”。
