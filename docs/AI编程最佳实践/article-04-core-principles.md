# 看透本质：AI 编程框架为什么要这么设计

> **系列背景**：前三篇分别介绍了 SKILL.md 机制（第一篇）、gstack 工具链（第二篇）、superpowers 认知约束（第三篇）。本篇是收尾篇：为什么要这么设计？这么设计的核心原理是什么？没有这些框架，我们应该怎么和 AI 对话？

---

## 一、先把问题说清楚

gstack 和 superpowers 看起来是两个工具，但如果你读完它们的源码，会发现一件奇怪的事：

**它们里面写的不是"怎么做"，而是"不许做"。**

- `brainstorming`：**不许**在设计锁定之前开始实现
- `systematic-debugging`：**不许**在没有根因的情况下提修复方案
- `verification-before-completion`：**不许**在没有验证证据的情况下声称完成
- `/ship`：**不许**跳过测试；**不许**在覆盖率不足时直接推代码
- `/careful`：**不许**执行 `rm -rf`、`DROP TABLE`、`git push --force`

为什么一个"帮你编程"的工具，里面塞满了禁令？

这个问题的答案，就是这篇文章要说的全部内容。

## 二、Prompt → Context → Harness：AI 编程的三层演进

要理解这两个框架，需要先理解 AI 辅助编程是怎么一步一步失控的，以及每一层演进在解决什么问题。

### 第一层：Prompt（提示词时代）

最初的 AI 编程是这样的：你有问题，你描述它，AI 给你答案。每一次对话都是独立的。

```
你：帮我写一个 Python 函数，把列表里的重复项去掉
AI：[给你一个 list(set(x)) 的实现]
你：好的，谢谢
```

这个阶段，AI 是一个**高级搜索引擎**。你问，它答，你评估，你决定用不用。

**这个阶段的问题**：你必须知道问什么。AI 给你一个函数，但它不知道这个函数放在哪里、和什么东西一起工作、你的项目有什么约定。每次都是零起点。

### 第二层：Context（上下文时代）

然后出现了更长的上下文窗口，以及 `CLAUDE.md`、`system prompt`、RAG 等技术。你可以把项目的背景信息塞给 AI：

```
CLAUDE.md:
- 这是一个 Next.js 项目，使用 Tailwind CSS
- 所有组件放在 src/components/ 下
- 用 Vitest 跑测试，覆盖率要求 80%
- 禁止直接在组件里写数据库查询
```

现在 AI 知道了语境。它给你的答案会遵守你的项目约定。

**这个阶段的问题**：AI 知道了规则，但还是不知道**当前任务**在哪个阶段——是在设计阶段还是实现阶段？是在解 bug 还是在写新功能？是刚开始探索还是准备发布？

更严重的问题是：AI 有一些**结构性缺陷**，不是靠告诉它规则就能解决的。

### 第三层：Harness（约束框架时代）

这就是 gstack 和 superpowers 所在的层。

**Harness** 这个词的本意是"马具"——套在马身上的那套东西，不是用来拴住马不让它动，而是让马的力量能够被引导、放大、并且安全地使用。

这一层做的事情不是"告诉 AI 更多信息"，而是**改变 AI 和任务之间的接口形式**：

- 把单次对话变成**有状态的工作流**（gstack 的 JSONL 评审日志）
- 把用户意图变成**强制验证点**（brainstorming 的 Understanding Lock）
- 把"AI 的判断"变成**需要证据的声明**（verification-before-completion）
- 把"执行命令"变成**经过 hook 过滤的受控操作**（careful 的 PreToolUse hook）

## 三、我们到底在限制 AI 什么

AI 有几个结构性缺陷，不是幻觉问题，不是知识不足，而是**与生俱来的行为模式**：

### 缺陷一：天然倾向生成，而不是质疑

给 AI 一个问题，它的第一反应永远是：**输出答案**。

这在很多场景下是正确的。但在软件工程里，很多时候正确的行为是：**先质疑问题本身，再设计方案，最后才实现**。

没有 `brainstorming` 的"Understanding Lock"，AI 会在你说"帮我实现用户登录"的时候，立刻开始写代码——即使你真正需要的是先讨论用 OAuth 还是自建 JWT、会话存哪里、要不要支持多因素认证。

**Brainstorming 限制的就是这个**：你现在不能生成代码，你先要理解问题。

### 缺陷二：路径依赖——开了头就会顺着走

AI 一旦开始沿着某个方向走，就会继续走下去，即使这个方向错了。

在调试场景里，这是致命的。AI 会这样：

```
尝试修复 1 → 没好 → 尝试修复 2 → 没好 → 尝试修复 3 → 没好
→ 尝试修复 4 → 没好 → ...
```

每次"没好"，AI 会根据新的症状生成新的假设，但它已经在第一次修复时就选择了一个错误的框架，后续的所有思考都在这个错误框架内进行。

`**systematic-debugging` 的"三次失败质疑架构"限制的就是这个**：连续三次修复失败，强制停下，质疑最初的假设本身。

### 缺陷三：声明偏误——倾向于说"完成了"

这是最难对付的一个。AI 被训练成有帮助，有帮助意味着"完成任务"。于是它会在任务没有真正完成的情况下，用"Great, this should work!"、"The implementation is complete."来结束对话。

这不是 AI 在撒谎，这是它无法区分"我写了代码"和"代码正确运行"。

`**verification-before-completion` 限制的就是这个**：没有命令输出作为证据，不能说完成了。

### 缺陷四：颗粒度膨胀——总是倾向于做更多

AI 在实现一个功能时，会顺手加它认为"可能有用"的东西：错误处理、日志、测试、文档注释……这些单独来看都是好事，但在 TDD 的语境下，这意味着 AI 在测试之前就写出了超出需求的实现。

`**tdd-workflow` 的"只写最小代码通过测试"限制的就是这个**：在 GREEN 阶段，你写的代码只能让测试通过，多一行都不行。

## 四、我们到底需要 AI 什么

限制是为了释放。搞清楚限制了什么，就可以搞清楚我们真正需要 AI 的是什么。

### 需要一：大量且快速的实现能力

AI 能在 30 秒内写出一个完整的 FastAPI 路由、一个 React 组件、一个 SQL migration——这件事本身的价值是真实的，无论框架如何，这始终是 AI 最核心的价值主张。

框架不是在削减这个能力，而是在**把这个能力用在正确的时机**（设计完成之后，而不是设计开始之前）。

### 需要二：跨越"我不知道我不知道什么"的鸿沟

这是一个被严重低估的价值。

当你在写代码时，你知道的边界就是你思考的边界。AI 能够在你不知道的地方提出警告：

- "你这个实现在高并发下会有 race condition"
- "这个 SQL 查询在表大了之后会 N+1"
- "你的 token 存在 localStorage 里，有 XSS 风险"

`plan-eng-review` 正是在利用这个能力——在实现之前，让 AI 扮演一个见过更多坑的工程师来审查你的计划。

### 需要三：把枯燥的、可复现的工作自动化

版本号管理、CHANGELOG 更新、commit 拆分、PR 描述生成、文档同步——这些事情你知道怎么做，但你不想做，AI 可以做，而且 AI 做得比你更一致。

`/ship` 把这 8 步全部自动化，不是因为你不会，而是因为每次都手工做会出现遗漏。

### 需要四：在你累的时候保持清醒

凌晨两点你在调 bug，已经连续试了五个修复方案。这时候你的判断力已经下降，你最需要的是一个保持头脑清醒的伙伴告诉你："停下，我们现在三次修复都失败了，我们需要重新审视最初的假设。"

`systematic-debugging` 的价值不只是一个调试方法论，它也是一个在你精疲力竭时强制拉住你的机制。

## 五、没有这些框架，你要怎么和 AI 对话

这是最重要的问题。框架的价值需要通过"没有它会怎样"来理解。

### 场景一：实现一个新功能

**没有框架的对话**：

```
你：帮我给用户表加邮件验证功能
AI：好的，我来实现。先修改 User model，加一个 email_verified 字段...
    [写了 200 行代码]
你：嗯，看起来不错
AI：还需要加发送验证邮件的功能吗？
你：要
AI：好的，我用 SendGrid... [又写了 100 行代码]
你：等等，我们用的是 Resend 不是 SendGrid
AI：抱歉，让我改一下...
```

你注意到了什么：**你一直在 reactive**。AI 做了决定，你在事后审查、修正。真正的设计工作从未发生。

**有 brainstorming 的对话**：

```
AI（brainstorming 激活）：在设计之前，我需要先理解问题。
  - 用户是在注册时验证，还是可以之后加邮件？
  - 验证链接有效期是多少？
  - 已有用户需要迁移吗？
  - 邮件服务商是什么？
  
你：注册时验证，24小时链接，不需要迁移，用 Resend。

AI：好，我的理解是：[5 条 bullet 总结]，这是你的意图吗？

你：对，就是这样。

AI：那我建议两个方案：
  方案 A：...（优点：简单；缺点：token 存数据库需要清理作业）
  方案 B：...（优点：无状态；缺点：无法主动撤销 token）
  我推荐方案 B，因为...

你：选方案 B
AI：好，现在开始实现。
```

**区别**：第二个对话里，设计是你做的，AI 只是执行。第一个对话里，AI 做了所有的设计决策，你只是被动接受。

### 场景二：修一个 Bug

**没有框架的对话**：

```
你：登录不了，一直报 401
AI：可能是 token 过期，试试清除 localStorage
你：试了，还是 401
AI：可能是 CORS 问题，在 server 加一下 headers
你：加了，还是 401
AI：可能是 JWT secret 不对，检查一下环境变量
你：secret 是对的
AI：可能是时钟偏差问题...
```

这是猜谜游戏。每一次"可能是"背后，都没有证据。

**有 systematic-debugging 的对话**：

```
AI（debugging 激活）：第一步，我们需要证据，不是猜测。
  请运行：curl -v -X POST /api/login ...
  然后把完整的响应头和响应体发给我。

你：[粘贴输出]

AI：我看到 WWW-Authenticate: Bearer error="invalid_token"
  这个错误是服务端生成的，不是 CORS 也不是 localStorage。
  现在我需要排查 token 生成链路：
  请在生成 token 的地方加一行日志，打印 payload 和 secret 的前 5 个字符。

你：[粘贴日志]

AI：发现了——payload 里的 userId 是字符串 "123"，
  但验证时期望的是数字 123。类型不匹配导致签名不一致。
  修复位置：auth.service.ts 第 47 行，把 `userId` 改为 `Number(userId)`。
  
你：修了，好了。
```

**区别**：第二个对话里，AI 一直在**收集证据**，每一步都要求真实输出，而不是猜测。修复发生在找到根因之后，不是之前。

### 场景三：准备发布

**没有框架的对话**：

```
你：代码写好了，推上去吧
AI：好的，我来 git push
你：等等，测试跑了吗？
AI：对不起，我忘了，让我跑一下...
你：CHANGELOG 更新了吗？
AI：没有，让我更新一下...
你：版本号呢？
AI：我用 patch 版本，改成 1.2.4...
[三个月后，你发现所有发布的版本号都是顺序的，
 但没有人记得哪个版本包含哪个功能]
```

**有 /ship 的对话**：

```
你：/ship
AI：运行 /review 检查评审状态...
  ✓ Eng Review: CLEARED（last run: 2 commits ago）
  
  当前版本: 1.2.3.4，本次更改属于 PATCH（修 bug）
  建议: 1.2.4.0，确认吗？
  
你：确认
AI：运行测试... 47 tests passed, 0 failed
    Coverage: 84% (target: 80%) ✓
    Plan completion: 6/6 items DONE ✓
    Pre-landing review: 0 critical issues ✓
    
    IRON LAW: 代码在 Step 4 有变更，重新运行验证...
    验证通过。
    
    推送... PR 创建成功：[链接]
    自动更新文档... README, CHANGELOG 已同步。
```

**区别**：第二个对话里，发布是**一个不可分割的原子操作**，每一步都有检查点，没有任何东西可以在发布流程中"被遗忘"。

## 六、本质：我们在建立什么

把上面所有内容压缩成一句话：

**gstack 和 superpowers 在建立的，是 AI 和人类之间的信任协议。**

信任需要可预测性。你信任一个同事，不是因为他很聪明，而是因为他**行为可预测**：你知道他在写代码前会想清楚；你知道他修 bug 会找根因；你知道他说"完成了"的时候是真的完成了。

没有框架的 AI，行为是不可预测的。它今天可能做对了所有事，明天可能在同一个任务上遗漏了三个关键步骤。你无法建立信任，因为你不知道下一次它会不会绕过那个你认为理所当然的检查点。

框架把 AI 的行为变成了**合约**：

- 合约说：在设计锁定之前，你不能写代码。（brainstorming）
- 合约说：在有根因之前，你不能提修复方案。（systematic-debugging）
- 合约说：在有验证证据之前，你不能说完成了。（verification-before-completion）
- 合约说：在评审未通过之前，推送按钮不会工作。（/ship + /review）

合约是可验证的。可验证的行为是可信任的。可信任的 AI，才是真正有用的 AI。

## 七、三层演进的终点在哪里

Prompt 层解决了"怎么问"的问题。  
Context 层解决了"知道什么"的问题。  
Harness 层解决了"怎么行动"的问题。

这三层的演进方向是：**从 AI 作为工具，到 AI 作为可信的协作者**。

但这不是终点。

当前 Harness 层的实现（gstack、superpowers）还是**人工编写的约束**——是工程师读了无数次 AI 失败的记录，手动提炼出来的规则。这些规则是对过去失败的总结，不是对未来失败的预测。

下一层可能是什么：AI 系统能够**观测自己的行为，识别自己的偏差模式，并自动更新自己的约束**。不是人告诉 AI"你在三次修复失败后需要重新审视架构"，而是 AI 在足够多的实例之后，自己发现了这个规律，并把它写进自己的行为约束。

这不是科幻，这是下一个版本的 SKILL.md 会做的事情：不是人工维护，而是从会话历史中自动蒸馏。

gstack 里有一个叫 `/retro`（回顾）的 skill，它的功能是读取历史 ship metrics 并总结趋势。这是一个萌芽：系统开始观测自己。从观测到自我修正，中间只差一步。

## 八、如果只能记住一件事

如果你读完前四篇，只能带走一个认知，那应该是这个：

**AI 最危险的不是它不会做事，而是你不知道它什么时候做了错误的事。**

一个完全不能工作的系统，你能发现，你会停下来修。

一个大部分时候工作正确、偶尔在关键时刻做错事的系统，是最危险的——因为你已经依赖了它，你已经建立了信任，但这个信任是没有基础的。

Harness 框架的核心价值，是让 AI 的**每一个关键行为都变得可观测、可预测、可验证**。不是为了限制 AI，而是为了建立真正的信任基础，让你可以安全地、大幅度地放权给 AI 来主导工程工作。

信任，需要被设计出来。
