# 团队工作流标准化：把 AI 实践变成工程规范

> **系列背景**：前四篇从机制（SKILL.md）、工具链（gstack）、认知约束（superpowers）、底层原理出发，理解了 AI 编程框架为什么要这么设计。本篇是实践篇：如何在真实团队里，让这些框架真正落地，并且引导团队产生属于自己的框架。

---

## 一、为什么团队需要自己的框架

读完前四篇，你可能会有一个冲动：把 gstack 和 superpowers 直接发给团队，说"以后就这么用"。

这会失败。

不是因为这两个框架不好，而是因为**任何外部框架都是别人痛点的结晶，不是你团队痛点的结晶**。

gstack 解决的是 Garry Tan 团队的问题：发布流程容易被跳过、评审结果没有持久化、浏览器测试是手工的、设计质量容易退化。这些问题你们团队可能也有，也可能没有。

superpowers 解决的是 Anthropic 工程师在使用 Claude Code 时发现的问题：AI 容易在设计未完成时开始实现、容易在没有根因时提修复方案、容易在未验证时声称完成。这些问题你们团队可能感受很强烈，也可能感受不到。

框架必须从**你们自己的痛点**生长出来，才会有人真正使用它。

---

## 二、框架是怎么生长出来的

### 第一步：不要制定规范，先记录失败

在引入任何 AI 工作规范之前，先花两周时间做一件事：**记录所有"AI 让我们吃过的亏"**。

在团队的日会或周会里，加一个 5 分钟的环节：

```
【AI 工作复盘】
- 这周有没有因为 AI 输出导致的问题？
- 如果有，具体是什么？（不是"AI 写错了"，而是"AI 在___场景下做了___，导致___"）
- 你有没有重复问 AI 同一类型的问题超过 3 次？
- 有没有哪个 AI 回答特别好，但你不知道为什么？
```

两周后，你会得到一份真实的失败清单。这份清单里隐藏着你们团队需要的框架。

**常见的失败模式**（根据你们记录来验证）：

| 失败描述 | 背后的问题 | 对应的约束设计 |
|---------|-----------|--------------|
| AI 写的代码和我们的代码风格完全不一样 | 没有 Context 注入 | CLAUDE.md / 项目级 SKILL.md |
| 让 AI 改了 A，结果它顺手改了 B | 没有范围约束 | `/freeze` 机制 |
| AI 说"完成了"，但实际上没测试 | 无验证门控 | verification-before-completion |
| 反复跟 AI 解释同一个业务背景 | 无持久化上下文 | 业务领域 context 文档 |
| AI 的方案每次都不一样，无法预测 | 无固定 workflow | team SKILL.md |
| 产品写了需求，AI 理解成另一回事 | PRD 格式不利于 AI 消费 | AI 友好的 PRD 模板 |

### 第二步：把重复问题变成 SKILL.md

当你们找到一个**重复出现**的 AI 交互问题之后，动作是：

1. 用一句话描述这个问题（"AI 在___时会___"）
2. 设计一个约束（"遇到___时，AI 必须先___，才能___"）
3. 把这个约束写成 SKILL.md
4. 在接下来两周里，每次触发这个场景时测试约束是否有效

**关键原则**：不要在第一次出现问题时就写 SKILL.md。等它出现第三次以上，你才真正理解了这个问题的模式。

### 第三步：版本化你们的框架

框架不是一次写完的，它会演化。把你们的团队 SKILL.md 放进代码仓库，像对待代码一样对待它：

```
.team-ai/
├── SKILL.md                    # 全局约束（所有人所有任务都适用）
├── product/
│   └── SKILL.md                # 产品线约束（产品经理用）
├── engineering/
│   └── SKILL.md                # 工程约束（开发用）
├── qa/
│   └── SKILL.md                # 测试约束（测试用）
└── CHANGELOG.md                # 框架变更记录（每个约束的来源和原因）
```

每次修改 SKILL.md，都要在 CHANGELOG.md 里写清楚：**这个约束是因为什么失败而加的**。这是框架有生命力的关键——团队成员能够追溯每一条规则的来源，而不是把它当作不解释的权威。

## 三、产品线：AI 辅助 PRD 工作流

产品经理和 AI 的交互，最核心的问题是：**PRD 是写给人看的，AI 不擅长消费**。

传统 PRD 的格式（背景 → 需求描述 → 功能列表 → 原型图说明）是为了让其他人类理解意图而设计的，其中包含大量隐含信息和上下文。AI 在读这类文档时，会丢失大量信息，或者做出错误的推断。

### 3.1 PRD 的 AI 友好化改造

不是颠覆现有 PRD 格式，而是在 PRD 末尾加一个"AI 消费块"：

```markdown
## AI 消费块（机器可读段）

### 前置条件（这些已经存在，不需要实现）
- 用户系统已有邮件字段，未验证
- 邮件服务：Resend，API Key 在环境变量 RESEND_API_KEY

### 本次变更的边界（只做这些，其余不动）
- 注册流程：注册后发送验证邮件
- 登录限制：邮件未验证的用户，登录成功后跳转到提醒页
- 提醒页：展示"请验证邮件"，提供重发验证邮件功能

### 不做什么（明确排除）
- 不做"修改邮件地址"功能
- 不做验证状态的管理后台
- 不做已有用户的批量迁移

### 验收标准（每条可被测试验证）
- [ ] 新注册用户收到验证邮件
- [ ] 点击邮件链接，状态变为"已验证"
- [ ] 验证链接 24 小时后失效
- [ ] 未验证用户登录后看到提醒页
- [ ] 提醒页"重发邮件"按钮可用，限流 1 次/分钟

### 依赖（实现前需要确认）
- Resend 是否支持自定义发件人域名？（需 @infrastructure 确认）
```

**这个格式的设计原理**：
- **前置条件**：让 AI 不要重新实现已有的东西
- **边界**：让 AI 不要"顺手"扩展功能
- **不做什么**：显式排除，比只写"做什么"更清晰
- **验收标准**：每条都是可被 AI 或测试人员验证的断言

### 3.2 产品 AI 工作流的 SKILL.md 模板

```yaml
---
name: product-prd
description: |
  产品需求文档撰写规范。当需要撰写、评审或拆解 PRD 时使用。
  确保 PRD 包含 AI 消费块，验收标准可被自动化验证。
---

# 产品 AI 工作规范

## PRD 撰写规范

在输出任何 PRD 之前，必须回答以下问题：

1. **这个需求解决了谁的什么问题？** 一句话，不超过 50 字。
2. **成功的样子是什么？** 至少 3 条可被验证的验收标准。
3. **不做什么？** 至少列出 2 条明确排除的功能。
4. **这个需求依赖什么已有系统？** 列出所有前置条件。

如果以上任何一项回答不了，PRD 不可以输出。

## 需求评审规范

在评审 PRD 时，逐条检查：
- 验收标准是否都是可测试的断言？（"用户体验好"不是验收标准）
- 是否有范围界定？（没有"不做什么"的 PRD 很危险）
- 开发友好：是否有"AI 消费块"格式？

## 与开发的交接规范

产品输出 PRD 后，在 Kickoff 会议里，与开发一起完成：
- 开发用 AI 读 PRD，输出理解摘要
- 产品确认摘要是否准确
- 不一致的地方立即在 PRD 里修正，而不是口头约定
```

### 3.3 引导产品团队的步骤

**第一周**：不改变任何流程，只是在每个 PRD 末尾加上"AI 消费块"。让开发把这个块给 Claude 读，看 Claude 的理解是否和产品意图一致，把不一致的地方记录下来。

**第二周**：根据记录的不一致，优化 AI 消费块的格式——哪些字段总是"被 AI 误读"，就说明描述方式需要改。

**第三周**：把改进后的格式写成 SKILL.md，团队开始统一使用。

**一个月后**：用一个下午的时间回顾所有"开发说需求没理解清楚"的情况，看使用新格式前后是否有改善。

## 四、工程线：AI 辅助开发工作流

开发和 AI 的交互最复杂，因为开发的工作跨越多个阶段：理解需求 → 设计方案 → 实现 → 测试 → 发布。每个阶段 AI 的行为模式都不同，需要的约束也不同。

### 4.1 从 PRD 到代码：需求分解工作流

拿到 PRD 之后，不要直接开始写代码，先用 AI 做需求分解：

**第一步：让 AI 读 PRD，输出它的理解**

```
提示词模板（可写入 SKILL.md）：

请阅读这份 PRD，用以下格式输出你的理解：

1. 核心功能（3 条以内，每条一句话）
2. 涉及的系统组件（列出所有需要修改的文件/模块）
3. 你认为不确定或有歧义的地方（如果有）
4. 你认为 PRD 没有提到但可能需要考虑的技术问题（如果有）

不要开始实现，只输出理解。
```

产品和开发一起确认 AI 的理解是否正确，不一致的地方在 PRD 里修正。

**第二步：让 AI 生成实现计划（不是代码）**

```
提示词模板：

基于上面确认的理解，请生成实现计划：

格式要求：
1. 每个步骤是一个原子变更（一个函数/一个迁移/一个组件）
2. 每个步骤包含：做什么 → 验证：如何确认这步正确 ← 依赖：需要前面哪步完成
3. 按照依赖顺序排列

不要写代码，只写计划。
```

让计划先经过一次人工的快速评估（5 分钟），确认顺序合理，没有明显遗漏。

**第三步：按步骤逐一实现**

每个步骤独立完成，完成后立即验证（运行测试/检查输出），再进入下一步。不要让 AI 一口气实现所有步骤。

### 4.2 开发 SKILL.md 模板

```yaml
---
name: engineering-dev
description: |
  开发实现规范。在实现任何功能时使用。
  确保实现有计划、按步骤执行，每步有验证。
---

# 开发 AI 工作规范

## 实现前必做

在开始写任何代码之前：

1. **读项目约定**：检查 CLAUDE.md 和 .team-ai/engineering/SKILL.md
2. **拆分步骤**：把任务拆成原子步骤，每步不超过 50 行代码变更
3. **确认理解**：用一句话描述这步要做什么，如果说不清楚，步骤拆得不够细

禁止：把整个功能作为一步让 AI 实现。

## 实现中规范

- 每步实现完成，立即运行相关测试，确认通过再继续
- 如果测试失败：先找根因，不要直接让 AI 修复（见 systematic-debugging）
- 如果 AI 在实现 A 的时候"顺手"改了 B：回滚 B 的修改，把 B 记录为下一步

## 代码审查规范

提交前，让 AI 做一次快速自检：

```
请检查刚才的实现：
1. 有没有 console.log / print 遗留？
2. 有没有硬编码的配置值？（IP、URL、密钥）
3. 有没有未处理的错误情况？
4. 有没有超出 PRD 范围的改动？

只列出问题，不要修复。
```

## 禁止事项

- 禁止直接执行 `rm -rf` 相关命令（须确认）
- 禁止 `git push --force`
- 禁止在没有测试覆盖的情况下修改核心业务逻辑
- 禁止在未经产品确认的情况下修改 PRD 范围外的功能
```

### 4.3 团队 Git 工作流与 AI 的结合

把 AI 引入 Git 工作流，不是"让 AI 写 commit message"，而是建立提交纪律：

**Commit 粒度规范**（写入 SKILL.md）：

```markdown
每个 commit 必须对应 PRD 里的一个验收标准，或者一个实现计划里的一个步骤。

commit message 格式：
type(scope): 描述

type:
- feat: 实现新功能
- fix: 修复 bug  
- test: 添加/修改测试
- refactor: 重构（不改变行为）
- docs: 文档

body（可选）：
- 对应 PRD 验收标准: #AC-3
- 如有，说明为什么这么实现而不是其他方式

禁止：在一个 commit 里混合多种类型的变更
```

**PR 描述规范**（让 AI 生成初稿）：

```
让 AI 根据 commit 列表生成 PR 描述：

请根据以下 commit 列表，生成 PR 描述：
[粘贴 git log --oneline]

格式要求：
1. 一句话总结这个 PR 做了什么
2. 对应的 PRD 验收标准：逐条列出，标注 DONE / NOT APPLICABLE
3. 测试：说明测试了哪些场景（不是测试覆盖率）
4. 风险：列出任何可能影响现有功能的地方
```

## 五、测试线：AI 辅助测试工作流

测试和 AI 的关系最容易被误解。

常见的误解：**让 AI 生成测试用例**。这是把 AI 当成测试用例生成器，价值有限，而且容易产生大量低质量的"代码覆盖率刷量用例"（测试逻辑存在但没有真正验证行为）。

正确的思维框架：**让 AI 帮你发现你没有想到的场景**。

### 5.1 测试的 AI 工作流：从 PRD 生成测试矩阵

**不是让 AI 直接写测试代码，而是先生成"测试矩阵"**：

```
提示词模板（写入 qa/SKILL.md）：

请阅读这份 PRD（附上 PRD），生成测试矩阵。

格式：

| 场景 | 前置条件 | 操作 | 预期结果 | 类型 |
|------|---------|------|---------|------|

类型分类：
- Happy Path（正常路径）
- Edge Case（边界情况）
- Error Case（错误情况）
- Security（安全场景）
- Performance（性能场景）

要求：
1. Happy Path 至少 3 条
2. Edge Case 至少 5 条（登录时：空邮箱、邮箱格式错误、密码错误等）
3. Security 场景：考虑越权访问、注入、重放攻击
4. 不要生成测试代码，只生成矩阵

完成后，标注你认为"最容易被开发遗漏"的 3 个场景，说明理由。
```

**测试矩阵的作用**：不是为了让 AI 写代码，是为了让测试人员和开发人员在早期就对齐"什么算完成"，同时发现 PRD 里的空白。

### 5.2 AI 辅助探索性测试

对于已经上线的功能，用 AI 协助做探索性测试——类似 gstack `/qa` 的思路，但不依赖任何工具，只依赖测试人员和 AI 的对话：

**探索性测试提示框架**：

```
我要对 [功能名称] 做探索性测试。

已知信息：
- 功能描述：[一句话]
- 技术栈：[前端/后端/数据库]
- 用户角色：[管理员/普通用户/未登录用户]

请帮我设计探索路径：
1. 从"正常用户的完整使用流程"开始，列出 5 个关键节点
2. 对每个节点，列出可以"斜着走"的方向（什么样的异常操作可能暴露问题）
3. 特别关注：状态切换（已登录↔未登录，有权限↔无权限）

格式：探索地图（文字版，不需要代码）
```

测试人员用这个地图指导测试，发现问题后记录在 Bug 报告里，同时记录"AI 预测的场景里，哪些确实发现了问题"——这个数据帮助你优化提示词。

### 5.3 测试 SKILL.md 模板

```yaml
---
name: qa-testing
description: |
  测试工作规范。在设计测试方案、执行测试、记录 Bug 时使用。
  确保测试覆盖 PRD 验收标准，同时发现 PRD 未覆盖的场景。
---

# 测试 AI 工作规范

## 测试设计规范

在开始任何测试之前：

1. **读 PRD 验收标准**：逐条转化为测试用例
2. **生成测试矩阵**：用 AI 补充 PRD 没有覆盖的边界场景
3. **标注优先级**：P0（阻塞发布）/ P1（希望修复）/ P2（可接受）

## Bug 报告规范

AI 辅助生成 Bug 报告草稿：

```
请帮我生成 Bug 报告草稿：

现象：[我观察到什么]
复现步骤：[我怎么触发的]
期望：[应该是什么]
实际：[实际是什么]
截图/日志：[附件]

请补充：
- 可能影响的相关功能
- 这个问题可能的根因方向（供开发参考）
- 建议的优先级（P0/P1/P2），说明理由
```

## 验收测试规范

每次发布前，针对本次 PRD 的所有验收标准做一次完整的验收测试，输出表格：

| 验收标准 | 状态 | 测试方法 | 备注 |
|---------|------|---------|------|
| 新注册用户收到验证邮件 | PASS | 手动注册 + 查邮箱 | |
| 验证链接 24h 失效 | PASS | 用过期 token 访问 | |
| 未验证用户看到提醒页 | FAIL | 登录后直接进入首页 | Bug #123 |

这个表格和 PR 一起提交，作为发布的依据。
```

## 六、把三条线连起来：团队 AI 协作协议

产品、开发、测试三条线分别有自己的 SKILL.md，但还需要一个跨线的协议，让三条线的 AI 工作流能够无缝衔接。

### 6.1 信息接力协议

```
PRD（产品）
  ↓ 包含 AI 消费块
Kickoff（产品 + 开发）
  ↓ AI 输出理解摘要，双方确认
实现计划（开发）
  ↓ 每步对应一个 PRD 验收标准
代码（开发）
  ↓ 每个 commit 对应一个实现步骤
测试矩阵（测试）
  ↓ 覆盖 PRD 验收标准 + AI 生成的边界场景
验收测试结果（测试）
  ↓ 表格形式，逐条标注 PASS/FAIL
发布决策（产品 + 开发 + 测试）
```

这条链上的每一个"↓"，都是 AI 帮助格式化和确认的节点，也是人类做决策的节点。

**关键设计原则**：AI 处理格式化、摘要、补充，人类做确认和决策。

### 6.2 团队全局 SKILL.md

```yaml
---

# 团队 AI 全局规范

## 永远适用的规则

### 信息传递规则
1. 跨角色传递的信息（PRD、计划、Bug 报告），必须用结构化格式
2. 口头确认的决定，必须在对应文档中更新，不能只存在于会话里
3. 使用 AI 生成的任何文档，发送前必须人工确认过一遍

### AI 使用范围
可以用 AI：
- 生成草稿（PRD、计划、报告、代码、测试用例）
- 检查完整性（有没有遗漏的场景、验收标准是否可测试）
- 理解摘要（把长文档压缩成要点）
- 格式转换（把非结构化信息转成结构化格式）

不应该用 AI 做最终决策：
- 发布决策（AI 可以提供信息，但发布 yes/no 是人来决定）
- 优先级决策（业务权衡不是 AI 擅长的）
- 法律/合规判断（需要专业人员）

### 失败记录规则
每周，每个人花 3 分钟记录一条"AI 工作中遇到的问题"：
- 格式：[角色] 在 [场景] 下，AI [做了什么]，导致 [什么问题]
- 记录位置：.team-ai/FAILURES.md

每月一次，从 FAILURES.md 里提炼新的约束，更新各角色 SKILL.md。
```

## 七、引导团队的节奏

不要在一个月里把所有东西推给团队。按照这个节奏：

### 第 1 个月：建立记录习惯

**目标**：团队开始系统地记录 AI 工作中的问题。

**行动**：
- 在每周例会里加 5 分钟的 AI 工作复盘
- 建立 `.team-ai/FAILURES.md` 文件，开始记录

**不要做的**：不要在这个阶段推任何约束，只是记录。

### 第 2 个月：第一个 SKILL.md

**目标**：根据第一个月的记录，解决频率最高的一个问题。

**行动**：
- 找出出现频率最高的失败模式
- 设计一个约束（一条规则）
- 把这一条规则写成团队的第一个 SKILL.md
- 在团队里演示如何使用，让每个人试一次

**不要做的**：不要写超过 5 条规则的 SKILL.md，一条够了。

### 第 3 个月：角色分化

**目标**：产品、开发、测试各自有一个专属的 SKILL.md。

**行动**：
- 每个角色各自提出自己最需要的一条约束
- 各角色的 leader 负责写这一条约束
- 全团队 review，合并进各自的 SKILL.md

### 第 4 个月起：迭代与演化

**目标**：SKILL.md 变成一个活的文档，每月迭代一次。

**行动**：
- 每月一次，从 FAILURES.md 里提炼新约束
- 删除不再需要的约束（团队已经内化了，就不需要写在文档里了）
- 把最有效的约束分享给其他团队

## 八、如何知道框架真的在生效

三个指标：

### 指标 1：同一类型的 AI 错误不再复现

如果某个 SKILL.md 约束是有效的，它对应的那类失败应该不再出现。如果还在出现，约束没有解决根本问题，需要重设计。

**怎么测量**：每月看 FAILURES.md，统计每类问题出现的次数。

### 指标 2："需要解释给 AI 的背景"越来越少

框架越完善，你需要在每次对话里手动解释的项目背景就越少——因为它们都写在 SKILL.md 里了。

**怎么测量**：主观感受，但可以具体到"这周有多少次，你发现自己在重复解释同一件事"。

### 指标 3：新成员能在一天内达到 AI 使用效率

如果框架好，新成员读完 SKILL.md，就能以接近其他人的效率使用 AI。

**怎么测量**：新成员 onboarding 后第一天的 AI 工作质量，和第一周末对比。

## 结语：框架的本质是凝固的团队经验

gstack 是 Garry Tan 团队凝固在代码里的经验。  
superpowers 是 Anthropic 工程师凝固在 SKILL.md 里的经验。  

你们团队的 SKILL.md，是你们自己的痛点和解法凝固的产物。

它不会在第一天就完美，它会在第一个月里有点生硬，在第三个月开始顺手，在第六个月时你会发现自己几乎不再想它——因为它已经成为团队工作方式的一部分，就像代码风格规范一样，你不再意识到它的存在，但你确实在遵守它。

这就是工程规范应该有的生命轨迹。
