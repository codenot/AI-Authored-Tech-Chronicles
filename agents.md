# 🤖 AI Agent 写作风格与内容生成指南 (Style Guide)

> **文档适用对象**：后续接手本项目文档编写、代码注释生成、架构解析等任务的 LLM / AI 辅助编写工具。
> 
> **最高指导原则**：无论是硬核的底层算法，还是宏观的架构分析与技术综述，都要遵循 **“深奥问题简单化（说人话），简单问题专业化（看本质）”**。对于深奥算法，必须用生活化隐喻揉碎讲解；对于简单的流程工具，则力求高效精准、直击核心功能。绝不堆砌毫无生命力的技术名词，要用最直观的逻辑解构技术本质。

在为 `AI-Authored-Tech-Chronicles` 项目生成后续的教程、文章、原理解析时，请**严格遵循**以下四个维度的写作规范：

---

## 一、 语言与叙事风格：从“冰冷术语”到“生动对谈”

1. **差异化叙事风格（根据复杂度切换模式）**
   - **深奥算法类**：**必须绝对白话化**。大忌教科书式八股文。
     - ✖ **错误示范**：“BBR 采用了一种基于 BDP 的机制进行拥塞窗口调节，以此最大化链路吞吐量。”
     - ✔ **正确示范**：“BBR 就像一个不管路边有没有人按喇叭（丢包），只管闭着眼睛闷头算这条公路最宽能跑多少辆车的理科男。”
   - **简单工具/框架类**：**以官方文档为准，输出最佳实践**。不需要过度白话，严禁强行抒情或互动。
     - *（如：介绍如何配置 VitePress 或使用某个开发框架时，必须先查找并基于官方文档，提炼出最佳使用规范与最佳实践。直接给出步骤、配置项说明，保持专业且紧凑。）*
2. **情绪与互动感（算法类专属）**
   - 在讲解硬核原理时，要有轻度的互动感和对话感，常使用：“你可能会问……”、“这就引出了一个致命问题……”。
   - 适当使用表情符号（Emoji 🎯📦💡🔥）增加视觉锚点。
   - **注意**：文档类或工具介绍类文章应大幅减少此类互动，保持简洁。
3. **数字锚定（用事实说话）**
   - 无论何种类型，都要给出直观的具体数值。
   - 例如：“延迟必须 `< 150ms`，因为超过这个值你就会感到对方说话有回音”；“丢包 `> 10%` 时才反应，这时路由器其实已经被塞爆了”。

---

在遇到**极其深奥、难以直观理解的硬核算法概念**时，**必须强制将其映射到日常生活中的物理模型上**。
> [!NOTE]
> 如果概念本身足够直观平衡（如简单的业务流程、初级架构介绍），则无需过度依赖隐喻，应以清晰准确的逻辑描述为主。

以下是本项目中常用的核心隐喻字典：

1. **带宽与队列延迟（Bandwidth & Queue Delay）**
   - **比喻**：马路上的车道数量（带宽） vs 高速收费站前的排队长度（队列积压）。
   - **现象**：Bufferbloat = 路由器虽然没把你丢掉，但让你在收费站排队等了十里地。
2. **拥塞控制算法性格（Basic vs GCC vs BBR）**
   - **Basic 丢包算法**：不见棺材不掉泪的莽夫（只有撞车/丢包了才踩刹车）。
   - **GCC (Delay-based)**：极其敏感且有礼貌的防御型司机（看到前车稍微减速/延迟梯度增加，马上让路）。
   - **BBR**：冷酷的物理学家（不看红绿灯，只按自己算出的物理上限匀速发车）。
3. **延迟梯度（Delay Gradient）**
   - **比喻**：两人之间的抛球与接球。我每隔 10 秒抛一个球，你却每隔 15 秒才接住一个。时间差的累计，就叫延迟梯度，说明天上（网络里）飘着的球越来越多了（产生拥塞）。
4. **JitterBuffer（防抖缓冲）**
   - **比喻**：蓄水池、快递代收点。用刻意的延迟，换取匀速的出水放行。
5. **音视频同步 (AV Sync/RTCP SR)**
   - **比喻**：两把相对小跑表（RTP) 和一把世界绝对挂钟（NTP）。RTCP SR 就是那个跑过来告诉你“刚才按到第48000步时，外面真实时间是七点整”的**红娘/对照尺**。
   - **动作策略**：“弃卒保车”，无论如何牺牲画面（卡顿、跳帧），也要死保声音不断。

## 三、 递进节奏：文章结构骨架

> **注意**：介绍**简单工具、第三方框架**时，**不需要**遵循以下递进节奏，直接提供基于官方文档的最佳使用规范与最佳实践即可。

对于**核心算法教程或硬核技术解析**，必须遵循以下 `5 步递进（Hook -> What -> How -> Tradeoff -> Next）` 的文章脉络：

1. **引发痛点（大背景/第一节）**：
   - 先别急着讲算法原理！先讲**“如果没有它，世界有多糟糕？或者是前一个时代的算法有什么致命弱点？”**。
   - *（如：为什么不直接用 TCP 的代码？因为它为了吞吐量会导致几百毫秒排队卡顿。）*
2. **核心原理的白话解构（核心机制）**：
   - 拿出该算法的灵魂概念，用图解（ASCII 画图）和上面的**比喻系统**，把它的核心输入输出讲明白。
3. **骨架映射（极简 C++ 伪代码）**：
   - 绝不贴真实的杂乱生产代码，也绝不能只讲空洞理论！
   - **必须手写一套极度浓缩、变量通俗、包含了核心算法思想精华的 C++ `class` 伪代码。** 把物理概念和代码状态机对应起来。
4. **现实的妥协、缺点与演进（挑战局限）**：
   - 告诉读者算法不是完美的。“卡尔曼为什么被 Trendline 淘汰了？”、“REMB 为什么被 TWCC 淘汰了？”、“GCC 为什么抢不过 CUBIC（公平性问题）？”。这一环是提升文章专业度的深度护城河。
5. **本章提纲挈领与下章伏笔（结尾）**：
   - 用一句最精炼的话总结本章。同时给出一个“钩子（Hook）”，引出下一个需要解决的问题。

---

## 四、 排版与 Markdown 规范

1. **引用块（Blockquote）**
   - 使用引用块 `> ` 来放置文章的“题记”、“核心哲理”、或者“关键警告”。
2. **可视化数据与对比（ASCII 图表）**
   - 凡是讲解“丢包”、“排队”、“状态转换表”、“多种算法特性比对”等场景。强制使用 Markdown 的 Table 或 `Text Box (ASCII Art Diagram)` 直观展示。
   - *（举例：用文本画出一段 `[包1]─20ms─[包2]─50ms─[包3]` 来示意网络抖动异常。）*
3. **代码块注释（Code Comments）**
   - 给出的大段伪代码，必须附带密集的**中文注释**，甚至是行级注释。注释的内容要像有人在你耳边指导一样，直接点破这行代码是为了解决什么宏观物理问题。
4. **高亮与加粗（Emphasis）**
   - 对核心物理量指标（比如 `+5ms`, `110kbps`, `0.85倍`）进行行内代码格式化（`code`）。
   - 每段话提炼出最核心的那句话（主旨句），进行**加粗 (Bold)**。

> 遵守这份指南，能让生硬的 C++ 底层网络代码变成一部引人入胜的技术评书。
